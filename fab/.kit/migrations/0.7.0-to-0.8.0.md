# Migration: 0.7.0 to 0.8.0

## Summary

Restructure `fab/config.yaml`: extract `context:` to `fab/context.md`, extract `code_quality:` to `fab/code-quality.md`, remove dead `stages:` section, and add `model_tiers:` section (absorbing `fab/.kit/model-tiers.yaml`). Updates the config.yaml header comment to reference new companion files.

## Pre-check

- `fab/config.yaml` exists and contains at least one of: `context:` section, `stages:` section, or lacks a `model_tiers:` section

## Changes

### 1. Extract `context:` to `fab/context.md`

If `fab/config.yaml` contains a `context:` key:

1. Read the `context:` value (typically a YAML multi-line string)
2. Create `fab/context.md` with the content:
   ```markdown
   # Project Context

   {extracted context value}
   ```
3. Remove the entire `context:` section from `fab/config.yaml` — including the section header comment block (from `# context —` through the `context: |` line and its indented value)

If `fab/context.md` already exists: skip extraction, still remove the `context:` section from config.yaml.

If `fab/config.yaml` has no `context:` section: skip this step entirely.

### 2. Extract `code_quality:` to `fab/code-quality.md`

If `fab/config.yaml` contains an **active** (uncommented) `code_quality:` section:

1. Read the `code_quality:` value — extract `principles`, `anti_patterns`, and `test_strategy` sub-keys
2. Create `fab/code-quality.md`:
   ```markdown
   # Code Quality

   ## Principles

   {each principle as a bullet: - {value}}

   ## Anti-Patterns

   {each anti_pattern as a bullet: - {value}}

   ## Test Strategy

   {test_strategy value, default: test-alongside}
   ```
3. Remove the `code_quality:` section from `fab/config.yaml`

If `fab/config.yaml` contains only **commented-out** `code_quality:`:

1. Copy `fab/.kit/scaffold/code-quality.md` to `fab/code-quality.md`
2. Remove the commented-out `code_quality:` block from `fab/config.yaml` (from `# code_quality —` through the last `#   test_strategy:` line)

If `fab/code-quality.md` already exists: skip creation, still remove the `code_quality:` section/comments from config.yaml.

If `fab/config.yaml` has neither active nor commented `code_quality:`: skip this step entirely.

### 3. Remove `stages:` section

If `fab/config.yaml` contains a `stages:` key:

1. Remove the entire `stages:` section — including the section header comment block (from `# stages —` through all stage entries ending before the next top-level section)

If no `stages:` section exists: skip.

### 4. Add `model_tiers:` section

If `fab/config.yaml` does NOT contain a `model_tiers:` key:

1. Add the following section at the end of the file (before any trailing whitespace):
   ```yaml
   # model_tiers — Maps provider-agnostic tier names to provider-specific model
   #   identifiers. Used by fab-sync.sh when deploying skills to agent directories.
   # Consumed by: fab/.kit/sync/2-sync-workspace.sh (agent file generation).
   #
   # Tiers:
   #   fast    — simple skills (script delegation, lookups, formatted output)
   #   capable — complex skills (artifact generation, reasoning, code analysis)
   #
   # null = use platform default (no model: override in generated agent file).
   model_tiers:
     fast:
       claude: haiku
     capable:
       claude: null    # null = use platform default
   ```

If `model_tiers:` already exists: skip.

### 5. Update config.yaml header comment

Replace the companion files comment block in `fab/config.yaml`:

**Before** (any of these patterns):
```yaml
# Companion files (not configured here):
#   fab/constitution.md  — project principles & constraints (MUST/SHOULD rules)
#   docs/memory/index.md   — centralized documentation index
```

**After**:
```yaml
# Companion files (not configured here):
#   fab/constitution.md    — project principles & constraints (MUST/SHOULD rules)
#   fab/context.md         — free-form project context (tech stack, conventions)
#   fab/code-quality.md    — coding standards for apply/review (optional)
#   docs/memory/index.md   — centralized documentation index
```

If the header doesn't match the expected pattern: skip (don't break custom headers).

## Verification

- `fab/context.md` exists (or no `context:` was present in the original config)
- `fab/config.yaml` does not contain a `context:` key (active or in section comments)
- `fab/config.yaml` does not contain a `stages:` key
- `fab/config.yaml` does not contain `code_quality:` (active or commented)
- `fab/config.yaml` contains a `model_tiers:` key
- `fab/config.yaml` header references `fab/context.md` and `fab/code-quality.md`
