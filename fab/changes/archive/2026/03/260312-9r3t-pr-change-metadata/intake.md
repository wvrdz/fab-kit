# Intake: PR Change Metadata

**Change**: 260312-9r3t-pr-change-metadata
**Created**: 2026-03-12
**Status**: Draft

## Origin

> User requested adding change identity information (change ID, change name, Linear issue links) to the PR description generated by `/git-pr`. Discussion explored placement options and Linear URL construction strategy.

## Why

PR descriptions generated by `/git-pr` currently show pipeline quality metrics (Stats table) but lack change identity — the fab change ID, human-readable name, and linked Linear issues. This makes it harder to trace a PR back to its fab change or associated issue tracker entry without manually cross-referencing `.status.yaml` or the branch name.

Adding this metadata improves traceability between GitHub PRs, fab changes, and Linear issues — especially useful when reviewing PRs or searching for the change that produced a given PR.

## What Changes

### 1. New "Change" section in PR body template

Add a **"Change"** section to the `/git-pr` PR body, placed **above** the existing "Stats" section. This section contains a table with three columns:

```markdown
## Change
| ID | Name | Issue |
|----|------|-------|
| r3m7 | add-oauth-support | [DEV-123](https://linear.app/weaver-ai/issue/DEV-123) |
```

- **ID**: From `.status.yaml` `id` field (4-char). Show `—` if unavailable.
- **Name**: From `.status.yaml` `name` field (full folder name). Show `—` if unavailable.
- **Issue**: From `fab status get-issues <change>` (already fetched in Step 1). Each issue ID rendered as a Linear link using `https://linear.app/{linear_workspace}/issue/{ISSUE_ID}`. Multiple issues comma-separated. If `linear_workspace` is not configured, show bare issue IDs. Show `—` if no issues.

The entire "Change" section is **conditional on `{has_fab}`** — if there's no fab change, omit it entirely (same gating as the pipeline progress line).

### 2. New `linear_workspace` field in `fab/project/config.yaml`

Add an optional `linear_workspace` field to the project config:

```yaml
project:
  name: "fab-kit"
  description: "FAB Kit — Tooling for Specification-Driven Development"
  linear_workspace: "weaver-ai"  # optional — enables Linear issue links in PRs
```

This field is:
- **Optional** — if absent, `/git-pr` falls back to showing bare issue IDs (e.g., `DEV-123`) without hyperlinks
- **Project-level** — set once per project during `/fab-setup` or manually
- **Used only by `/git-pr`** for URL construction: `https://linear.app/{linear_workspace}/issue/{ISSUE_ID}`

### 3. Update `/git-pr` skill to read `linear_workspace`

The skill already loads `fab/project/config.yaml` as part of its context. It needs to:
1. Read `linear_workspace` from config (may be absent)
2. Read `.status.yaml` `id` and `name` fields (already resolved via `fab change resolve`)
3. Use the already-fetched `issues` from Step 1
4. Construct the "Change" table with conditional link formatting

### 4. Migration for discoverability

Add a migration file (range from last migration target to the release version shipping this change). The migration:

1. Check if `fab/project/config.yaml` already has a `linear_workspace` field — if so, skip
2. Add a commented-out `linear_workspace` field under the `project:` block:
   ```yaml
   project:
     name: "..."
     description: "..."
     # linear_workspace: "your-workspace"  # optional — enables Linear issue links in PRs
   ```
3. Print a note explaining the new field and how to enable it

This is a discoverability migration — it doesn't change behavior, just surfaces the new option to existing users during `/fab-setup migrations`.
