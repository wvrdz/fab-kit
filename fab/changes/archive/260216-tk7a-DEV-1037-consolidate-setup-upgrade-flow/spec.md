## Non-Goals

- Changing the internal logic of `sync-workspace.sh` beyond renaming/relocating and adding stale cleanup — the script's core behavior (directory creation, symlinks, scaffold files, VERSION handling) stays the same. The one allowed addition is a stale artifact cleanup pass
- Changing the migration file format or discovery algorithm — only the skill that runs them changes name
- Adding new subcommands to `/fab-setup` beyond the four defined (bare, `config`, `constitution`, `migrations`)
- Updating spec files (`docs/specs/`) — those are updated during hydrate, not apply

## Scripts: Promote `sync-workspace.sh` to `fab-sync.sh`

### Requirement: Relocate Script

`fab/.kit/scripts/lib/sync-workspace.sh` SHALL be moved to `fab/.kit/scripts/fab-sync.sh`. The script becomes a first-class user-facing entry point alongside `fab-upgrade.sh` and `fab-help.sh`.

#### Scenario: Script relocation

- **GIVEN** `fab/.kit/scripts/lib/sync-workspace.sh` exists at its current location
- **WHEN** this change is applied
- **THEN** the script is moved to `fab/.kit/scripts/fab-sync.sh`
- **AND** `fab/.kit/scripts/lib/sync-workspace.sh` no longer exists

### Requirement: Preserve Script Behavior

The script's internal logic SHALL NOT change. Only the file path changes. All existing functionality (directory creation, `fab/VERSION` handling, skill symlinks, agent files, `.envrc`, `.gitignore` entries, scaffold file copying) MUST continue to work identically.

#### Scenario: Identical output after relocation

- **GIVEN** `fab-sync.sh` at its new location
- **WHEN** a user runs `fab/.kit/scripts/fab-sync.sh`
- **THEN** the output and side effects are identical to running the old `lib/sync-workspace.sh`

### Requirement: Internal Path Updates

The script's internal `SCRIPT_DIR` / path resolution logic SHALL be updated to account for its new location (one directory level up from before). Any relative path calculations within the script that depend on `$SCRIPT_DIR` MUST be adjusted.

#### Scenario: Script resolves kit root correctly

- **GIVEN** `fab-sync.sh` lives in `fab/.kit/scripts/` (not `fab/.kit/scripts/lib/`)
- **WHEN** the script calculates paths to `fab/.kit/templates/`, `fab/.kit/scaffold/`, or `fab/.kit/VERSION`
- **THEN** all paths resolve correctly

### Requirement: Update Test Infrastructure

The test directory `src/lib/sync-workspace/` SHALL be renamed to `src/lib/fab-sync/`. The symlink, README, and test files within MUST be updated to reference `fab-sync.sh` at its new path.

#### Scenario: Test suite continues to pass

- **GIVEN** the test directory is renamed to `src/lib/fab-sync/`
- **WHEN** the test suite runs
- **THEN** all existing tests pass against `fab-sync.sh` at its new path

### Requirement: Callers Updated

All scripts that invoke `sync-workspace.sh` MUST be updated to call `fab-sync.sh` at the new path:
- `fab-upgrade.sh` — calls sync after extracting new kit
- Any other internal references

#### Scenario: fab-upgrade.sh calls fab-sync.sh

- **GIVEN** `fab-upgrade.sh` previously called `lib/sync-workspace.sh`
- **WHEN** this change is applied
- **THEN** `fab-upgrade.sh` calls `fab-sync.sh` (sibling in same `scripts/` directory)

### Requirement: Stale Artifact Cleanup

`fab-sync.sh` SHALL include a stale artifact cleanup pass after creating/repairing symlinks and agent files. For each agent integration path (Claude Code, OpenCode, Codex), the script SHALL:

1. Scan the target directory for existing skill entries (symlink directories, flat files, agent files)
2. Compare against the current `skills[]` array (built from `.kit/skills/*.md`)
3. Remove any entries not in the current skills list

For Claude Code agent files (`.claude/agents/`), the cleanup SHALL only remove agent files that correspond to skills that no longer exist in `.kit/skills/` AND have `model_tier: fast` characteristics (i.e., were generated by sync). Agent files not generated by sync (e.g., user-created) SHOULD be preserved.

#### Scenario: Stale symlinks removed after skill deletion

- **GIVEN** `.claude/skills/fab-init/SKILL.md` exists as a broken symlink (fab-init.md was deleted)
- **AND** `.claude/agents/fab-init.md` exists from a previous sync run
- **WHEN** `fab-sync.sh` runs
- **THEN** `.claude/skills/fab-init/` directory is removed
- **AND** `.claude/agents/fab-init.md` is removed
- **AND** equivalent OpenCode and Codex stale entries are removed

#### Scenario: Non-sync agent files preserved

- **GIVEN** `.claude/agents/custom-agent.md` was created by the user (not by sync)
- **WHEN** `fab-sync.sh` runs its stale cleanup
- **THEN** `custom-agent.md` is preserved (it doesn't correspond to any `.kit/skills/*.md`)

## Skills: New `/fab-setup` Skill

### Requirement: Skill File Location and Metadata

A new skill file SHALL be created at `fab/.kit/skills/fab-setup.md` with frontmatter:

```yaml
---
name: fab-setup
description: "Set up a new project, manage config/constitution, or apply version migrations. Safe to re-run."
model_tier: fast
---
```

#### Scenario: Skill file exists with correct frontmatter

- **GIVEN** the change is applied
- **WHEN** an agent reads `fab/.kit/skills/fab-setup.md`
- **THEN** the file exists with `name: fab-setup`, `model_tier: fast`, and a description covering all four subcommands

### Requirement: Context Loading Exceptions

`/fab-setup` SHALL have subcommand-specific context loading behavior:

| Subcommand | Always Load (config + constitution) | Change Context |
|------------|-------------------------------------|----------------|
| *(bare)* | Skip if files don't exist (first-run) | Skip |
| `config` | Skip if files don't exist | Skip |
| `constitution` | Skip if files don't exist | Skip |
| `migrations` | Load config (MUST exist for migrations) | Skip |

This combines the exceptions from `/fab-init` (skips Always Load on first run) and `/fab-update` (loads config but skips change context). The `migrations` subcommand requires `config.yaml` to exist because migrations may reference project configuration.
<!-- clarified: Context loading varies by subcommand — bootstrap/config/constitution are lenient, migrations requires config -->

#### Scenario: Bare invocation without config

- **GIVEN** `config.yaml` does not exist (first-time setup)
- **WHEN** `/fab-setup` is invoked with no arguments
- **THEN** the skill does NOT attempt to load config.yaml or constitution.md as context
- **AND** it proceeds to call `fab-sync.sh` and then create the files

#### Scenario: Migrations requires config

- **GIVEN** `config.yaml` does not exist
- **WHEN** `/fab-setup migrations` is invoked
- **THEN** the skill loads `config.yaml` as part of its context
- **AND** if `config.yaml` is missing, it stops with guidance to run `/fab-setup` first

### Requirement: Bare Invocation (Full Bootstrap)

When `/fab-setup` is invoked with no arguments, it SHALL perform the full bootstrap sequence:

1. Verify `fab/.kit/` and `fab/.kit/VERSION` exist (same pre-flight as current `/fab-init`)
2. Call `fab/.kit/scripts/fab-sync.sh` via Bash (structural setup)
3. If `config.yaml` doesn't exist → invoke config subcommand in create mode
4. If `constitution.md` doesn't exist → invoke constitution subcommand in create mode

This is the same behavior as current `/fab-init` with no arguments, updated to call `fab-sync.sh` instead of `lib/sync-workspace.sh`.

#### Scenario: First-time project setup

- **GIVEN** a project with `fab/.kit/` but no `config.yaml` or `constitution.md`
- **WHEN** the user runs `/fab-setup`
- **THEN** `fab-sync.sh` creates directories, symlinks, skeleton files, and `.gitignore` entries
- **AND** the user is prompted to create `config.yaml` interactively
- **AND** the user is prompted to create `constitution.md` interactively

#### Scenario: Re-run on existing project

- **GIVEN** a project with `config.yaml` and `constitution.md` already present
- **WHEN** the user runs `/fab-setup`
- **THEN** `fab-sync.sh` runs idempotently (repairs any missing directories/symlinks)
- **AND** config and constitution creation are skipped (files exist)
- **AND** output indicates the project is already set up

#### Scenario: Kit not found

- **GIVEN** `fab/.kit/` does not exist
- **WHEN** the user runs `/fab-setup`
- **THEN** the skill stops immediately with: `fab/.kit/ not found. Copy the kit directory into fab/.kit/ first — see the Getting Started guide.`

### Requirement: `config` Subcommand

`/fab-setup config [section]` SHALL behave identically to current `/fab-init config [section]`. All logic (section menu, interactive editing, targeted string replacement, post-edit validation) is preserved. The only change is the command name in user-facing messages and error output.

#### Scenario: Edit a config section

- **GIVEN** `config.yaml` exists
- **WHEN** the user runs `/fab-setup config context`
- **THEN** the `context` section is presented for editing
- **AND** after editing, validation runs automatically
- **AND** the user sees results referencing `/fab-setup`, not `/fab-init`

#### Scenario: Config creation mode

- **GIVEN** `config.yaml` does not exist
- **WHEN** the user runs `/fab-setup config`
- **THEN** a new `config.yaml` is generated interactively (same flow as `/fab-init config` in create mode)

### Requirement: `constitution` Subcommand

`/fab-setup constitution` SHALL behave identically to current `/fab-init constitution`. All logic (create mode, amendment mode, semantic versioning, structural rules) is preserved. The only change is the command name in user-facing messages.

#### Scenario: Amend constitution

- **GIVEN** `constitution.md` exists with version `1.1.0`
- **WHEN** the user runs `/fab-setup constitution` and adds a new principle
- **THEN** version bumps to `1.2.0`
- **AND** the amendment is applied
- **AND** output references `/fab-setup`, not `/fab-init`

### Requirement: `migrations` Subcommand

`/fab-setup migrations [file]` SHALL absorb all logic from the current `/fab-update` skill:

1. Compare `fab/VERSION` to `fab/.kit/VERSION`
2. Scan `fab/.kit/migrations/` for applicable files
3. Validate non-overlapping ranges
4. Apply migrations sequentially (sorted by FROM ascending)
5. Update `fab/VERSION` after each successful migration

The optional `[file]` argument allows targeting a specific migration file directly, bypassing version range discovery. When `[file]` is provided, the skill reads and applies that migration file without version comparison.

#### Scenario: Apply pending migrations

- **GIVEN** `fab/VERSION` is `0.2.0` and `fab/.kit/VERSION` is `0.4.0`
- **AND** `fab/.kit/migrations/0.2.0-to-0.3.0.md` and `0.3.0-to-0.4.0.md` exist
- **WHEN** the user runs `/fab-setup migrations`
- **THEN** both migrations are applied in order
- **AND** `fab/VERSION` is updated to `0.4.0`

#### Scenario: Already up to date

- **GIVEN** `fab/VERSION` equals `fab/.kit/VERSION`
- **WHEN** the user runs `/fab-setup migrations`
- **THEN** the skill reports "Already up to date" and takes no action

#### Scenario: Target specific migration file

- **GIVEN** the user has a specific migration to apply
- **WHEN** the user runs `/fab-setup migrations 0.3.0-to-0.4.0.md`
- **THEN** that specific migration is read and applied
- **AND** `fab/VERSION` is updated to `0.4.0`

#### Scenario: Missing fab/VERSION

- **GIVEN** `fab/VERSION` does not exist
- **WHEN** the user runs `/fab-setup migrations`
- **THEN** the skill stops with: `fab/VERSION not found. Run fab-sync.sh to create it.`
<!-- clarified: Error message references fab-sync.sh (which creates VERSION) instead of old /fab-init -->

#### Scenario: Migration failure mid-chain

- **GIVEN** `fab/VERSION` is `0.2.0` and migrations `0.2.0-to-0.3.0.md` and `0.3.0-to-0.4.0.md` exist
- **WHEN** the first migration succeeds but the second migration fails during application
- **THEN** `fab/VERSION` is updated to `0.3.0` (last successful migration)
- **AND** the skill stops immediately with the failure details
- **AND** output suggests: `Re-run /fab-setup migrations to retry from 0.3.0.`

#### Scenario: Overlapping migration ranges detected

- **GIVEN** `fab/.kit/migrations/` contains files with overlapping ranges (e.g., `0.2.0-to-0.4.0.md` and `0.3.0-to-0.5.0.md`)
- **WHEN** the user runs `/fab-setup migrations`
- **THEN** the skill aborts before applying any migrations
- **AND** reports: `Overlapping migration ranges detected. Kit may be corrupted.`

### Requirement: No `validate` Subcommand

`/fab-setup` SHALL NOT have a separate `validate` subcommand. Validation is folded into the `config` and `constitution` subcommands — each validates its own output after every create/edit operation. Passing `validate` as an argument SHALL produce: `Validation is built into /fab-setup config and /fab-setup constitution — each validates after every edit.`

#### Scenario: User tries /fab-setup validate

- **GIVEN** a user accustomed to `/fab-init validate`
- **WHEN** they run `/fab-setup validate`
- **THEN** they see a redirect message explaining validation is now built-in

### Requirement: Unrecognized Arguments

When arguments other than recognized subcommands (`config`, `constitution`, `migrations`) are passed, `/fab-setup` SHALL output: `Unknown subcommand: {arg}. Valid: config, constitution, migrations. Run /fab-setup with no arguments for full setup.`

#### Scenario: Unrecognized argument

- **GIVEN** a user runs `/fab-setup foobar`
- **WHEN** the argument is parsed
- **THEN** the user sees the unknown subcommand message listing valid options

## Deprecated Requirements

### `/fab-init` Skill

**Reason**: Consolidated into `/fab-setup`. The name "init" implied a one-time operation, but the skill also managed ongoing config/constitution editing. `/fab-setup` better communicates both initial setup and ongoing management.
**Migration**: Use `/fab-setup` (bare) for bootstrap, `/fab-setup config` for config management, `/fab-setup constitution` for constitution management.

### `/fab-update` Skill

**Reason**: Absorbed into `/fab-setup migrations`. The two-step flow (`fab-upgrade.sh` then `/fab-update`) caused dropped-ball failures — users would upgrade and forget to run migrations. Folding migrations into `/fab-setup` makes the post-upgrade step discoverable from the same command namespace.
**Migration**: Use `/fab-setup migrations` instead of `/fab-update`.

### `lib/sync-workspace.sh` Location

**Reason**: Promoted from internal plumbing (`scripts/lib/`) to user-facing script (`scripts/`). As a standalone entry point that solves the chicken-and-egg problem, it deserves first-class visibility alongside `fab-upgrade.sh` and `fab-help.sh`.
**Migration**: Use `fab/.kit/scripts/fab-sync.sh` instead of `fab/.kit/scripts/lib/sync-workspace.sh`.

### `/fab-init validate` Subcommand

**Reason**: Validation folded into `config` and `constitution` subcommands. Each path now validates its own output after every create/edit. A separate validate step was redundant — users either forgot to run it or ran it unnecessarily.
**Migration**: Validation is automatic after `/fab-setup config` and `/fab-setup constitution` edits.

## Design Decisions

1. **Two entry points, not one**: `/fab-setup` (LLM skill) and `fab-sync.sh` (bash script) remain separate.
   - *Why*: `fab-sync.sh` must be runnable without an LLM (chicken-and-egg — skills require symlinks that sync creates). Merging them would require either an LLM for structural setup or shell logic for interactive config.
   - *Rejected*: Single combined tool — would violate the chicken-and-egg constraint or require embedding LLM invocation in shell.

2. **`fab-sync.sh` in `scripts/` not `scripts/lib/`**: Promoted to user-facing directory.
   - *Why*: It's a user-facing entry point (users run it directly for repair, after bootstrap download). `lib/` is for internal plumbing that users never invoke directly.
   - *Rejected*: Keeping in `lib/` with documentation — hidden location contradicts first-class status.

3. **No separate `validate` subcommand**: Validation built into `config` and `constitution` flows.
   - *Why*: Reduces surface area. Validation after manual edits can be triggered by running `/fab-setup config` or `/fab-setup constitution` which validates on entry. The standalone validate was rarely useful — most users edited via the subcommands which already validate.
   - *Rejected*: Keeping validate as a third subcommand — adds surface area for marginal benefit.

4. **Single `model_tier: fast` for all subcommands**: Including `migrations`.
   - *Why*: Existing migrations are mechanical (move dirs, rename files, find-replace strings) with structured instructions and built-in pre-check/verification guardrails. One tier per skill avoids needing infrastructure for per-subcommand tier switching.
   - *Rejected*: Separate skill or higher tier for migrations — adds complexity. Can revisit if migration complexity increases.
