# Proposal: Simplify Stages and Entry Paths

**Change**: 260212-v5p2-simplify-stages-entry-paths
**Created**: 2026-02-12
**Status**: Draft

## Why

The current 6-stage pipeline has a hollow first stage ("brief") that is entered and completed atomically by `/fab-new` — unlike other stages where you sit, iterate, and advance. Additionally, three overlapping entry paths (`/fab-new`, `/fab-discuss` new-mode, `/fab-discuss` refine-mode) create unnecessary complexity, with `/fab-discuss` having two internal modes and duplicating `/fab-new`'s folder creation. This change simplifies both the stage model and the entry flow.

## What Changes

- **Drop "brief" as a formal pipeline stage** — the pipeline becomes 5 stages: spec, tasks, apply, review, archive. `brief.md` is still generated by `/fab-new` as the input artifact, but it's no longer tracked as a stage in `.status.yaml` progress.
- **Make `/fab-new` the single adaptive entry point** — remove the hard "max 3 questions" cap, use SRAD to determine how many questions are needed. When the description is clear, one-shot generation. When vague, conversational (absorbing discuss's interactive style).
- **Absorb `/fab-discuss`'s gap analysis into `/fab-new`** — before creating a folder, check if the change is already covered by existing mechanisms.
- **Remove `/fab-discuss` entirely** — delete the skill definition, remove from all design docs, glossary, flow diagrams, and skill catalog. Brainstorming without committing to a change is just normal conversation with the agent.
- **Update `/fab-clarify` to handle brief refinement** — since discuss's "refine mode" is gone, clarify absorbs the ability to refine `brief.md` when `stage: spec`.
- **Update `config.yaml` stages section** — remove the `brief` stage entry, update `spec` to have no prerequisites (it's now the first stage).
- **Update all downstream skills** — `/fab-continue`, `/fab-ff`, `/fab-fff`, `/fab-status` to reflect 5 stages and the absence of brief as a stage.
