# Spec: Simplify Stages and Entry Paths

**Change**: 260212-v5p2-simplify-stages-entry-paths
**Created**: 2026-02-12
**Affected docs**: `fab/docs/fab-workflow/planning-skills.md`, `fab/docs/fab-workflow/change-lifecycle.md`, `fab/docs/fab-workflow/clarify.md`, `fab/docs/fab-workflow/configuration.md`, `fab/docs/fab-workflow/templates.md`, `fab/docs/fab-workflow/context-loading.md`, `fab/docs/fab-workflow/kit-architecture.md`

## Stage Pipeline: 5-Stage Model

### Requirement: Remove Brief as a Pipeline Stage

The pipeline SHALL consist of 5 stages: `spec`, `tasks`, `apply`, `review`, `archive`. The `brief` stage SHALL be removed from the pipeline definition. `brief.md` SHALL continue to be generated by `/fab-new` as the input artifact that feeds into the spec stage, but it SHALL NOT be tracked as a stage in `.status.yaml` progress.

#### Scenario: New change created by /fab-new
- **GIVEN** a user runs `/fab-new` with a description
- **WHEN** the change folder and `brief.md` are created
- **THEN** `.status.yaml` progress SHALL contain only: `spec: active`, `tasks: pending`, `apply: pending`, `review: pending`, `archive: pending`
- **AND** `brief.md` SHALL exist in the change folder as the input artifact

#### Scenario: Stage numbering in display
- **GIVEN** a change exists with the 5-stage pipeline
- **WHEN** `/fab-status` or `/fab-switch` displays stage progress
- **THEN** stages SHALL be numbered 1-5: spec (1), tasks (2), apply (3), review (4), archive (5)

### Requirement: Update config.yaml Stages Section

The `stages` section in `fab/config.yaml` SHALL remove the `brief` entry. The `spec` stage SHALL have no `requires` prerequisite (it is now the first stage). All other stage entries SHALL remain unchanged except that any reference to `brief` in `requires` lists SHALL be removed.

#### Scenario: config.yaml after update
- **GIVEN** the updated `config.yaml`
- **WHEN** an agent reads the stages section
- **THEN** the first stage SHALL be `spec` with `generates: spec.md`, `required: true`, and no `requires` field
- **AND** `tasks` SHALL require `[spec]`
- **AND** no `brief` stage entry SHALL exist

## State Machine: Single Source of Truth

### Requirement: Remove the `stage:` Field from .status.yaml

The `stage:` field SHALL be removed from `.status.yaml`. The current stage SHALL be derived from the progress map by finding the entry with value `active`. This eliminates the redundancy between the `stage:` pointer and the per-stage progress values, which previously required skills to cross-reference both fields.

#### Scenario: Deriving current stage
- **GIVEN** a `.status.yaml` with `progress: {spec: done, tasks: active, apply: pending, ...}`
- **WHEN** any skill needs to determine the current stage
- **THEN** it SHALL scan the progress map and find the `active` entry (`tasks`)

#### Scenario: No active entry (all done)
- **GIVEN** a `.status.yaml` with all progress entries set to `done`
- **WHEN** any skill needs to determine the current stage
- **THEN** it SHALL treat the change as fully completed

### Requirement: State Vocabulary

The progress map SHALL use exactly four states:

| State | Meaning |
|-------|---------|
| `pending` | Not yet started — this stage is waiting for earlier stages to complete |
| `active` | Current stage — the "you are here" marker |
| `done` | Completed successfully |
| `failed` | Completed with failures requiring rework (review stage only) |

**Invariant**: Exactly one entry in the progress map SHALL be `active` at any time, except when all entries are `done` (change completed) or the change has been archived.

#### Scenario: Normal forward progression
- **GIVEN** a change with `progress: {spec: active, tasks: pending, ...}`
- **WHEN** spec generation completes
- **THEN** progress SHALL become `{spec: done, tasks: active, ...}`
- **AND** the transition SHALL be a two-write operation: current stage → `done`, next stage → `active`

#### Scenario: Review failure — automatic backward movement
- **GIVEN** a change with `progress: {spec: done, tasks: done, apply: done, review: active, archive: pending}`
- **WHEN** `/fab-review` determines the review has failed
- **THEN** `/fab-review` SHALL set `review: failed` AND `apply: active` in a single operation
- **AND** the `active` marker moves backward without destroying the `failed` state on review
- **AND** the user runs `/fab-apply` next to address the failures

#### Scenario: Reset to earlier stage
- **GIVEN** a change with `progress: {spec: done, tasks: done, apply: active, ...}`
- **WHEN** the user runs `/fab-continue spec` to reset
- **THEN** progress SHALL become `{spec: active, tasks: pending, apply: pending, review: pending, archive: pending}`

### Requirement: Update .status.yaml Template

The `.status.yaml` template (`fab/.kit/templates/status.yaml`) SHALL:
1. Remove the `stage:` field entirely
2. Remove `brief` from the progress map
3. Set the first stage (`spec`) to `active` and all others to `pending`

The template SHALL produce:

```yaml
name: {NAME}
created: {CREATED}
created_by: {CREATED_BY}
progress:
  spec: active
  tasks: pending
  apply: pending
  review: pending
  archive: pending
checklist:
  generated: false
  path: checklists/quality.md
  completed: 0
  total: 0
confidence:
  certain: 0
  confident: 0
  tentative: 0
  unresolved: 0
  score: 5.0
last_updated: {CREATED}
```

#### Scenario: Template used by /fab-new
- **GIVEN** the updated status template
- **WHEN** `/fab-new` creates a new change
- **THEN** `.status.yaml` SHALL have no `stage:` field, and progress keys: `spec: active`, `tasks: pending`, `apply: pending`, `review: pending`, `archive: pending`

### Requirement: Preflight Script Derives Current Stage

`fab-preflight.sh` SHALL derive the current stage from the progress map (find the `active` entry) and output it as `stage:` in its YAML output. This provides backward compatibility — skills that parse preflight output for `stage:` continue to work without changes to their parsing logic.

#### Scenario: Preflight output
- **GIVEN** a `.status.yaml` with `progress: {spec: done, tasks: active, ...}`
- **WHEN** `fab-preflight.sh` runs
- **THEN** the output SHALL include `stage: tasks` (derived from the `active` entry)
- **AND** the output SHALL include the full progress map

#### Scenario: Preflight with all stages done
- **GIVEN** a `.status.yaml` with all progress entries `done`
- **WHEN** `fab-preflight.sh` runs
- **THEN** the output SHALL include `stage: archive` (or the last `done` entry)

## Planning Skills: Adaptive /fab-new

### Requirement: SRAD-Driven Question Count

`/fab-new` SHALL use SRAD scoring to determine how many questions to ask, replacing the fixed "max 3 questions" cap. When all decisions score Confident or Certain, `/fab-new` SHALL generate the brief in one shot with no questions. When Unresolved decisions exist, `/fab-new` SHALL ask questions for each, prioritized by lowest Reversibility + lowest Agent Competence.

#### Scenario: Clear description — no questions
- **GIVEN** a user runs `/fab-new "add documentation_accuracy to checklist categories"`
- **WHEN** SRAD scoring finds all decisions are Certain or Confident
- **THEN** `/fab-new` SHALL generate `brief.md` without asking any questions

#### Scenario: Vague description — multiple questions
- **GIVEN** a user runs `/fab-new "improve auth"`
- **WHEN** SRAD scoring identifies 4 Unresolved decisions
- **THEN** `/fab-new` SHALL ask all 4 questions (no fixed cap)
- **AND** questions SHALL be ordered by lowest Reversibility + lowest Agent Competence first

### Requirement: Conversational Mode for Vague Inputs

When the user's description has low Signal Strength across many decision points, `/fab-new` SHALL enter a conversational mode — asking questions one at a time, building on previous answers, similar to the former `/fab-discuss` interactive style. The conversation SHALL continue until all Unresolved decisions are resolved or the user signals satisfaction.

#### Scenario: Conversational flow
- **GIVEN** a user runs `/fab-new "make things faster"`
- **WHEN** SRAD scoring identifies low Signal Strength with 5+ Unresolved decisions
- **THEN** `/fab-new` SHALL ask questions one at a time, each building on the previous answer
- **AND** the user MAY end the conversation early by signaling satisfaction

#### Scenario: Clear description skips conversational mode
- **GIVEN** a user runs `/fab-new "rename checklist category from security to security_audit"`
- **WHEN** SRAD scoring identifies 0 Unresolved decisions
- **THEN** `/fab-new` SHALL generate `brief.md` in one shot without entering conversational mode

### Requirement: Gap Analysis Before Folder Creation

`/fab-new` SHALL perform gap analysis before creating the change folder. The analysis SHALL:
1. Check for existing mechanisms in the current workflow, codebase, or docs
2. Evaluate scope — is the idea too broad (should be split) or too narrow (part of something larger)?
3. Consider alternatives — simpler approaches, extending existing skills

If an existing mechanism covers the idea, `/fab-new` SHALL present its findings and let the user decide whether to proceed.

#### Scenario: Existing mechanism covers the idea
- **GIVEN** a user runs `/fab-new "add a way to refine briefs"`
- **WHEN** gap analysis identifies that `/fab-clarify` already handles brief refinement
- **THEN** `/fab-new` SHALL present its findings to the user
- **AND** the user SHALL decide whether to proceed with folder creation or cancel
- **AND** if the user cancels, no folder SHALL be created and no `Next:` line SHALL be shown

#### Scenario: No existing mechanism
- **GIVEN** a user runs `/fab-new "add parallel task execution"`
- **WHEN** gap analysis finds no existing mechanism
- **THEN** `/fab-new` SHALL proceed directly to brief generation

### Requirement: /fab-new Produces Only brief.md

`/fab-new` SHALL produce only `brief.md` as its output artifact. Unlike the former `/fab-discuss` (which produced both `brief.md` and `spec.md` in new-change mode), `/fab-new` SHALL NOT generate `spec.md`. The user runs `/fab-continue` or `/fab-ff` to generate the spec.
<!-- clarified: fab-new produces only brief.md — confirmed by user; raw input captured in brief's Origin section preserves signal for downstream spec generation -->

#### Scenario: Artifact output
- **GIVEN** a user runs `/fab-new` and completes the brief
- **WHEN** the change folder is created
- **THEN** the folder SHALL contain `.status.yaml`, `brief.md`, and the `checklists/` subdirectory
- **AND** `spec.md` SHALL NOT exist

### Requirement: Brief Template Includes Origin Section

The `brief.md` template SHALL include an **Origin** section that captures the user's raw input — the original description or prompt passed to `/fab-new`. This preserves the user's language and intent for downstream spec generation via `/fab-continue`, compensating for signal loss when the conversational context is compressed into structured brief sections.

#### Scenario: Brief generated from clear description
- **GIVEN** a user runs `/fab-new "add parallel task execution to fab-apply"`
- **WHEN** `brief.md` is generated
- **THEN** the Origin section SHALL contain the verbatim user description: "add parallel task execution to fab-apply"

#### Scenario: Brief generated from conversational mode
- **GIVEN** a user runs `/fab-new "improve auth"` and answers 4 clarifying questions
- **WHEN** `brief.md` is generated
- **THEN** the Origin section SHALL contain the initial description plus a summary of key decisions from the conversation (not the full transcript)

## Clarify: Brief Refinement Capability

### Requirement: Clarify Handles Brief Refinement When Current Stage is Spec

`/fab-clarify` SHALL support refining `brief.md` when the current stage is `spec` (i.e., `progress.spec` is `active` or `done`). Clarify's taxonomy scan SHALL include both `brief.md` and `spec.md` (if it exists), using **per-artifact categories**: brief taxonomy categories for `brief.md` and spec taxonomy categories for `spec.md`. This absorbs the "refine mode" previously provided by `/fab-discuss`.
<!-- clarified: per-artifact taxonomy categories — brief categories for brief.md, spec categories for spec.md; each taxonomy matches its artifact type -->

#### Scenario: Refining brief before spec is generated
- **GIVEN** a change with `progress.spec: active` and no `spec.md` yet
- **WHEN** the user runs `/fab-clarify`
- **THEN** clarify SHALL scan `brief.md` for gaps using brief taxonomy categories
- **AND** clarify SHALL allow the user to refine `brief.md` content

#### Scenario: Refining brief after spec is generated
- **GIVEN** a change with `progress.spec: done` and `spec.md` exists
- **WHEN** the user runs `/fab-clarify`
- **THEN** clarify SHALL scan both `brief.md` and `spec.md` for gaps
- **AND** changes to `brief.md` SHALL be permitted

### Requirement: Update Clarify Stage Guard

The clarify skill's stage guard SHALL accept `spec` and `tasks` as valid current stages (derived from the `active` entry in the progress map). Since `brief` is no longer a pipeline stage, it SHALL NOT appear in the guard's valid list. The guard behavior remains: if the current stage is `apply`, `review`, or `archive`, abort with suggestion to use `/fab-review`.

#### Scenario: Valid stages
- **GIVEN** a change with `progress.spec: active`
- **WHEN** the user runs `/fab-clarify`
- **THEN** clarify SHALL operate normally

#### Scenario: Post-planning rejection
- **GIVEN** a change with `progress.apply: active`
- **WHEN** the user runs `/fab-clarify`
- **THEN** clarify SHALL abort with suggestion to use `/fab-review`

## Downstream Skills: Pipeline Updates

### Requirement: Update /fab-continue Behavior

`/fab-continue` SHALL determine what to do by finding the `active` entry in the progress map:

| `active` entry | Action |
|---------------|--------|
| `spec` | Generate `spec.md` → set `spec: done`, `tasks: active` |
| `tasks` | Generate `tasks.md` + checklist → set `tasks: done`, `apply: active` |
| `apply` | Block: "Planning is complete. Run /fab-apply." |
| `review` or `archive` | Block: "Use /fab-review or /fab-archive." |

No `active` entry (all `done`) → Block: "Change is complete."

Reset flow SHALL accept `spec` or `tasks`. Reset to `brief` SHALL be rejected with: "Cannot reset to brief — brief is not a pipeline stage. Edit brief.md directly or use /fab-clarify."

#### Scenario: First continue after /fab-new
- **GIVEN** a change with `progress.spec: active`
- **WHEN** the user runs `/fab-continue`
- **THEN** `/fab-continue` SHALL generate `spec.md`
- **AND** update progress: `spec: done`, `tasks: active`

#### Scenario: Second continue generates tasks
- **GIVEN** a change with `progress: {spec: done, tasks: active, ...}`
- **WHEN** the user runs `/fab-continue`
- **THEN** `/fab-continue` SHALL generate `tasks.md`
- **AND** update progress: `tasks: done`, `apply: active`

#### Scenario: Reset to spec
- **GIVEN** a change with `progress: {spec: done, tasks: active, ...}`
- **WHEN** the user runs `/fab-continue spec`
- **THEN** progress SHALL become `{spec: active, tasks: pending, apply: pending, review: pending, archive: pending}`
- **AND** `spec.md` SHALL be regenerated

#### Scenario: Reset to brief rejected
- **GIVEN** a change at any stage
- **WHEN** the user runs `/fab-continue brief`
- **THEN** `/fab-continue` SHALL output: "Cannot reset to brief — brief is not a pipeline stage. Edit brief.md directly or use /fab-clarify."

### Requirement: Update /fab-ff Pipeline

`/fab-ff` SHALL fast-forward through `spec` and `tasks` (2 planning stages). The pipeline: `spec → auto-clarify → tasks → auto-clarify`. The brief is a prerequisite input, not a stage to generate. `/fab-ff` SHALL find the `active` entry and start from there, skipping stages already `done`.

#### Scenario: Fast-forward from new change
- **GIVEN** a change with `progress.spec: active` and `brief.md` exists
- **WHEN** the user runs `/fab-ff`
- **THEN** `/fab-ff` SHALL generate `spec.md`, run auto-clarify, generate `tasks.md`, run auto-clarify
- **AND** update progress: `spec: done`, `tasks: done`, `apply: active`

### Requirement: Update /fab-fff Gate

`/fab-fff` SHALL require `brief.md` to exist and `confidence.score >= 3.0`. The pipeline: `/fab-ff` → `/fab-apply` → `/fab-review` → `/fab-archive`. No changes to the gate logic beyond the brief-exists check replacing the brief-stage-done check.

#### Scenario: Pre-flight check
- **GIVEN** a change with `brief.md` and confidence score 4.6
- **WHEN** the user runs `/fab-fff`
- **THEN** `/fab-fff` SHALL proceed with the full pipeline

### Requirement: Update /fab-status Display

`/fab-status` SHALL display 5 stages numbered 1-5. The current stage SHALL be derived from the `active` entry in the progress map. The `brief` progress line SHALL be removed.

## Deprecated Requirements

### `/fab-discuss` Skill
**Reason**: Functionality absorbed into `/fab-new` (adaptive questioning, gap analysis) and `/fab-clarify` (brief refinement). Three overlapping entry paths created unnecessary complexity.
**Migration**: Use `/fab-new` for starting new changes (now adaptive). Use `/fab-clarify` at spec stage to refine briefs. Free-form brainstorming is normal conversation with the agent.

### `/fab-discuss` Context-Driven Mode Selection
**Reason**: With `/fab-discuss` removed, mode selection between new/refine is unnecessary.
**Migration**: `/fab-new` always creates new changes. `/fab-clarify` always refines existing artifacts.

### `/fab-discuss` Dual Artifact Output (brief + spec)
**Reason**: `/fab-new` produces only `brief.md`. The stage-by-stage flow preserves per-stage confidence scoring.
**Migration**: Run `/fab-new` for the brief, then `/fab-continue` or `/fab-ff` for the spec.

### Brief as Pipeline Stage
**Reason**: Brief was a hollow stage — entered and completed atomically by `/fab-new` with no iteration within the stage.
**Migration**: `brief.md` remains as an input artifact. Pipeline starts at `spec`. New changes begin with `progress.spec: active`.

### Fixed 3-Question Cap in `/fab-new`
**Reason**: SRAD scoring provides a principled, dynamic mechanism for determining question count.
**Migration**: SRAD-driven questioning — 0 questions for clear inputs, as many as needed for vague ones.

### `stage:` Field in .status.yaml
**Reason**: Redundant with the progress map. Created a two-source-of-truth problem where skills had to cross-reference both `stage:` and `progress.{stage}` to determine behavior. The `active` marker in the progress map serves as the single "you are here" pointer.
**Migration**: Remove `stage:` from `.status.yaml`. Derive current stage by finding the `active` entry in the progress map. `fab-preflight.sh` outputs a derived `stage:` field for skill backward compatibility.

## Design Decisions

### `/fab-new` as Single Adaptive Entry Point
**Decision**: Consolidate `/fab-new` and `/fab-discuss` into a single `/fab-new` that adapts its interaction style based on input clarity via SRAD scoring.
**Rationale**: Three overlapping entry paths (`/fab-new`, `/fab-discuss` new-mode, `/fab-discuss` refine-mode) created confusion about which to use. A single adaptive entry point is simpler to discover, document, and maintain. SRAD already provides the mechanism to vary interaction depth.
**Rejected alternatives**: Keeping both skills with clearer differentiation — still two entry points to explain. Renaming `/fab-discuss` to `/fab-explore` — naming doesn't fix the fundamental overlap.

### Brief Remains as Input Artifact, Not Pipeline Stage
**Decision**: Keep `brief.md` as the input artifact generated by `/fab-new`, but remove it from the stage pipeline. The pipeline starts at `spec`.
**Rationale**: The brief stage was hollow — atomically created and completed by `/fab-new`, never iterated within. Removing it from the pipeline eliminates a meaningless stage transition and simplifies stage counting from 6 to 5.
**Rejected alternatives**: Removing `brief.md` entirely — the brief serves as valuable context for spec generation. Making brief an iterative stage — adds complexity for minimal value since `/fab-clarify` handles refinement.

### `/fab-new` Produces Only brief.md
**Decision**: Unlike the former `/fab-discuss` which produced both `brief.md` and `spec.md`, `/fab-new` produces only `brief.md`.
**Rationale**: The stage-by-stage flow (brief → spec via `/fab-continue`) preserves clean separation of concerns and per-stage confidence scoring. Dual artifact output was a convenience of the conversational discuss flow; with adaptive `/fab-new`, the user can `/fab-ff` to reach spec quickly.
**Rejected alternatives**: `/fab-new` optionally producing spec in conversational mode — blurs the distinction between entry point and planning stages.

### Single Source of Truth: Progress Map with `active` Marker
**Decision**: Remove the `stage:` field from `.status.yaml`. The current stage is determined by finding the `active` entry in the progress map. The state vocabulary is `pending`, `active`, `done`, `failed`.
**Rationale**: The old model had two sources of truth — `stage: spec` AND `progress.spec: pending` — forcing skills to cross-reference both. This created a state matrix that was complex to reason about and prone to desynchronization. The `active` marker serves as an explicit "you are here" pointer within the progress map, supporting both forward progression and backward movement (e.g., review failure → back to apply). Unlike a "first pending" derivation, `active` handles non-linear flow without destroying done/failed states on other stages.
**Rejected alternatives**: Keeping `stage:` with simplified progress (still two sources of truth). Removing both `stage:` and `active`, using "first pending" derivation (cannot express backward movement after review failure). Removing `stage:` and `pending`, using only `active`/`done`/`failed` (loses the "not started" semantic for future stages).
