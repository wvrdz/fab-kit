# Archive Index

- **260212-a4bd-unify-fab-continue** — Unified `/fab-apply`, `/fab-review`, and `/fab-archive` into `/fab-continue`, reducing the developer command surface to two primary commands (`/fab-continue` + `/fab-clarify`). Extended reset flow to all 6 stages. Deleted standalone execution skill files and updated all cross-references across skills, docs, design specs, and README.
- **260212-egqa-switch-return-main** — Added `--blank` flag to `/fab-switch` for deactivating the current change (deletes `fab/current`). Orthogonal to `--branch` — compose with `--blank --branch main` to also switch git branches. Updated `change-lifecycle.md` with deactivation lifecycle and `/fab-switch` section.
- **260212-akhp-rename-fab-backfill** — Renamed `/fab-backfill` to `/fab-hydrate-design` for semantic consistency with the `/fab-hydrate` family. Pure rename — skill file, Claude skill directory, centralized doc, and all cross-cutting references updated. Archives preserved as historical records.
- **260212-4tw0-migrate-scripts-stageman** — Migrated `fab-status.sh` and `fab-preflight.sh` to source `stageman.sh` for schema-driven stage/state queries. Replaced all hardcoded stage lists, stage-number mappings, and state symbol functions with dynamic calls to the Stage Manager API. Added `validate_status_file` to preflight. Deduplicated documentation: consolidated `src/stageman/` to single README, moved `fab/.kit/schemas/README.md` to `fab/docs/fab-workflow/schemas.md`, fixed all dangling references. Fixed `get_state_suffix` awk bug for quoted strings with spaces.
- **260212-ipoe-checklist-folder-location** — Moved checklist from `checklists/quality.md` subfolder to `checklist.md` at change root, co-located with other artifacts. Renamed from `quality.md` to `checklist.md`. Updated all skill path references, `.status.yaml` template default, and centralized docs. No migration needed (no active changes had generated checklists).
- **260212-bk1n-rework-fab-ff-archive** — Extended `/fab-ff` from a planning-only command to a full pipeline (planning → apply → review → archive) with interactive clarification stops. Updated `/fab-fff` comparison table to clarify differentiation: fab-ff is "fast but interactive", fab-fff is "fully autonomous with confidence gate."
- **260212-29xv-scoring-formula** — The confidence scoring formula produces inflated scores (all 26 archived changes scored 4.1–5.0) due to the Confident penalty of 0.1 being too low. Increases the penalty to 0.3 for meaningful differentiation, and adds grade reclassification to `/fab-clarify` so scores increase after clarification.
- **260212-v5p2-brief-pipeline-stage** — Restores `brief` as a formal pipeline stage in `config.yaml` (6 stages: brief → spec → tasks → apply → review → archive). Fixes inconsistency between the `.status.yaml` template (which had `brief: active`) and config/skills (which didn't recognize brief). Updates fab-new to reference the template, fixes fab-continue error handling, updates fab-switch stage mapping to 6 stages, and corrects centralized docs. Preflight script tolerates legacy files missing the brief entry.
- **260212-k7m3-fix-consistency-drift** — Fixes 11 critical inconsistencies between design specs, centralized docs, and implementation: renames "Proposal" to "Brief" in templates, aligns DEFERRED timing, adds Origin section to design template, fixes confidence score default phrasing, updates slug word count, moves branch integration attribution to /fab-switch, updates .status.yaml schema in design, adds archive index maintenance docs, adds created_by to design examples, fixes backfill terminology, and replaces "proposal" references in _generation.md.
- **260212-emcb-clarify-fab-setup** — Unclear boundary between `/fab-init` and `fab-setup.sh` — both managed overlapping concerns. Clarified the delegation pattern (fab-setup.sh = structural, fab-init = interactive config), added `fab/design/` initialization to fab-setup.sh, and documented the split in centralized docs.
- **260212-0r8e-fix-created-by-github** — The current `.status.yaml` format captures `created_by` using `git config user.name`, which returns inconsistent values across configurations. Using GitHub IDs via `gh api user --jq .login` provides a stable, globally unique identifier with graceful fallback.
- **260212-r7k3-add-no-switch-flag** — `/fab-new` always activates the newly created change via `/fab-switch`. Added a `--no-switch` flag that skips activation, allowing change creation without disrupting the current active context.
- **260212-v5p2-simplify-stages-entry-paths** — Simplifies the Fab pipeline from 6 to 5 stages by removing "brief" as a formal stage, consolidates /fab-new and /fab-discuss into a single adaptive /fab-new with SRAD-driven questioning and gap analysis, removes /fab-discuss entirely, and introduces the `active` marker in the progress map as the single source of truth for current stage (replacing the redundant `stage:` field).
- **260211-r4w8-spec-template-sections** — When the plan stage was absorbed into the spec stage, two valuable sections were lost: Non-Goals (explicit scope boundaries) and Design Decisions (rationale and rejected alternatives). This adds both as optional sections to the spec template, with generation procedure updates and documentation.
- **260211-endg-add-created-by-field** — `.status.yaml` tracks when a change was created but not by whom. This adds a `created_by` field auto-detected from `git config user.name` at change creation time, displayed by `/fab-status`, with backward-compatible handling for older changes.
- **260210-k7p2-document-srad-framework** — SRAD is referenced by fab-fff confidence gating and the backlog as implemented, but the framework itself — what the acronym stands for, how scores are calculated, what thresholds gate fab-fff — is defined nowhere in the specs. This change creates a standalone `fab/specs/srad.md` covering the full framework and updates the specs index.
- **260210-h7r3-kit-distribution-update** — `fab/.kit/` is distributed by manual `cp -r` with no mechanism to pull updates. This change adds a script-based distribution system: `fab-update.sh` for atomic updates from GitHub Releases, `fab-release.sh` for packaging and publishing, a curl one-liner for bootstrapping new projects, and repo rename from `docs-sddr` to `fab-kit`.
- **260210-q7m3-reading-order-doc-map** — Newcomers face 50+ documentation files with no guidance on reading sequence. This adds a Documentation Map to README.md with three audience-specific reading paths (new user, contributor, spec reader), a grouped document inventory (Getting Started, Concepts, Reference, Internals), glossary links, and orientation notes in both index files.
- **260210-nan4-define-auto-mode-signaling** — `fab-ff` invokes `fab-clarify` in "auto mode" between stage generations, but no mechanism is defined for how one skill signals mode to another. This change defines an explicit `[AUTO-MODE]` prefix protocol in `_context.md` and updates `fab-clarify`, `fab-ff`, and centralized docs to use it.
- **260210-7wxx-add-specs-index-context-loading** — `_context.md` defines an "Always Load" protocol that includes `fab/specs/index.md`, but `fab-apply.md`, `fab-review.md`, and `fab-archive.md` omit it from their Context Loading sections. This adds the missing line to all three execution skills.
- **260210-3uej-fix-collision-handling-regenerate** — `fab-new.md` Step 2 collision handling instruction says to "append an additional random character" which produces a 5-character random component breaking the `{YYMMDD}-{XXXX}-{slug}` naming format. The correct behavior is to regenerate the entire 4-character component rather than appending to it.
- **260210-0p4e-fix-stage-guard-progress-check** — `fab-continue`'s stage guard checks the `stage` field name to determine whether to proceed, but does not account for the `progress` value of that stage. If a change has `stage: tasks` but `progress.tasks: active` (meaning task generation was interrupted mid-way), the guard sees "tasks" and blocks with "Planning is complete. Run /fab-apply", when it should allow resumption of the interrupted tasks generation.
- **260210-zr1f-discuss-auto-activate-when-no-current** — `/fab-discuss` creates a change folder but never activates it, requiring a manual `/fab-switch` even when no other change is active. This adds the conditional offer to set the new change as active (via internal `/fab-switch`) when `fab/current` is empty, eliminating unnecessary friction while preserving the safety net when another change is in progress.
- **260210-m3k7-multi-agent-support** — `fab-setup.sh` only created Claude Code symlinks despite the architecture being designed for multi-agent portability. This change extends the setup script to always create integrations for OpenCode (commands) and Codex (skills) alongside Claude Code, closing the gap between design intent and implementation.
- **260209-h3v7-fab-backfill** — Fab's data flow is one-directional (specs → changes → docs) with no mechanism to detect where docs have outgrown specs. `/fab-backfill` identifies structural gaps and proposes concise additions back to specs with interactive per-gap confirmation.
- **260209-r4w8-archive-index-longer-slugs** — As archived changes accumulate, there's no way to search or browse what was done without opening each folder individually. Expanding the slug length from 2-4 words to 2-6 words improves self-documentation across the archive, git branches, and file listings.
- **260208-lgd7-fab-discuss-command** — The workflow lacks a conversational entry point for exploring ideas and iteratively sharpening vague concepts into solid proposals, and `/fab-new` doesn't compute confidence scoring required by the `/fab-fff` gate.
- **260208-k3m7-add-fab-fff** — The current `fab-ff --auto` mode guesses everything and defers human interaction rather than eliminating it. A better model is to raise confidence high first via `/fab-clarify`, then run the full pipeline autonomously with a confidence score gate.
- **260208-4wg3-fix-hydrate-links** — Commit 9329bd5 moved fab spec files from `doc/fab-spec/` to `fab/specs/`, but three template reference links in `fab-hydrate.md` were not updated, breaking user navigation to template format documentation.
- **260207-sawf-fix-command-format** — All "Next:" command suggestions across fab files reference commands using the `/fab:xxx` colon format, but the actual Claude Code skill names use the `/fab-xxx` hyphen format, causing users to receive invalid command references.
- **260207-q7m3-separate-hydrate-smart-context** — `/fab-init` conflates structural bootstrap with doc hydration — two conceptually different operations — and once `fab/docs/` is properly populated, all fab skills should smart-load relevant documentation from indexed folders rather than relying on specs alone.
- **260207-m3qf-clarify-dual-modes** — `/fab-clarify` operates in a single autonomous mode where assumptions are resolved by the agent, but users should be the authority on ambiguity resolution, and `/fab-ff` skips clarification entirely, allowing gap assumptions to compound downstream.
- **260207-k5od-hydrate-generate-mode** — `/fab-hydrate` only handles ingesting external sources into `fab/docs/`, but many projects have undocumented behavior in source code that could be bootstrapped into documentation if users could point hydrate at their own codebase.
- **260207-cg03-standardize-script-names** — Scripts in `fab/.kit/scripts/` use bare names without a common prefix, making them inconsistent with the `fab-*` naming convention used for skills and harder to identify.
- **260207-bb1q-add-specs-index** — The fab workflow has `fab/docs/` for post-implementation truth but no persistent home for pre-implementation intent. Specs currently live only inside `fab/changes/` and get archived, making original design intent hard to find.
- **260207-5mjv-preflight-grep-scripts** — Every fab skill repeats the same pre-flight sequence of reading `fab/current`, validating directories, and loading configuration files. Consolidating this into a single script reduces duplication and provides a single validation point.
- **260207-09sj-autonomy-framework** — Planning skills make 38 out of 47 decision points as silent assumptions without surfacing what was decided or why, leaving high-blast-radius decisions unchecked and creating inconsistent interruption patterns across skills.
