**Change**: 260214-m3w7-formalize-assumptions-scoring
**Created**: 2026-02-14
**Affected memory**:
- `docs/memory/fab-workflow/planning-skills.md` (modify)

## Non-Goals

- Changing the confidence scoring formula itself (`5.0 - 0.3*confident - 1.0*tentative`, `unresolved > 0 → 0.0`) — only fixing the data that feeds it
- Migrating or rewriting existing change artifacts — old briefs/specs remain valid as-is
- Adding new SRAD dimensions or changing dimension weights

## SRAD Rules: Assumptions Table Structure

### Requirement: All Four Grades in Every Assumptions Table

Every `## Assumptions` table generated by a planning skill (`/fab-new`, `/fab-continue`, `/fab-ff`, `/fab-fff`) SHALL include rows for all four SRAD grades: Certain, Confident, Tentative, and Unresolved.

The previous rule excluding Certain and Unresolved from tables is deprecated. All grades MUST be recorded for complete state transfer between pipeline stages.

#### Scenario: Agent generates brief with mixed grades
- **GIVEN** `/fab-new` is generating a brief
- **WHEN** SRAD analysis produces 5 Certain, 2 Confident, 1 Tentative, and 1 Unresolved decision
- **THEN** the `## Assumptions` table SHALL contain 9 rows, one for each decision
- **AND** the summary line SHALL read `9 assumptions (5 certain, 2 confident, 1 tentative, 1 unresolved)`

#### Scenario: Agent generates brief with only Certain decisions
- **GIVEN** `/fab-new` processes a highly detailed, unambiguous description
- **WHEN** SRAD analysis classifies all decisions as Certain
- **THEN** the `## Assumptions` table SHALL contain one row per Certain decision
- **AND** the summary line SHALL read `{N} assumptions ({N} certain, 0 confident, 0 tentative, 0 unresolved)`

### Requirement: Required Scores Column

The `## Assumptions` table SHALL always include a `Scores` column as the fifth column with format `S:nn R:nn A:nn D:nn`, where each value is the 0–100 SRAD dimension score for that row.

The Scores column is mandatory for every row. There is no optional/detection mode.

#### Scenario: Every row includes dimension scores
- **GIVEN** a planning skill generates an `## Assumptions` table
- **WHEN** the table is written to an artifact
- **THEN** every data row SHALL have exactly 5 pipe-delimited columns: `| # | Grade | Decision | Rationale | Scores |`
- **AND** the Scores cell SHALL match the pattern `S:\d+ R:\d+ A:\d+ D:\d+`

### Requirement: Unresolved Status Context in Rationale

Every Unresolved row in an Assumptions table SHALL include status context in its Rationale column explaining what happened with the decision.

Status context MUST use one of these patterns:
- `Asked — {outcome}` (e.g., "Asked — user undecided", "Asked — needs team input")
- `Deferred — {reason}` (e.g., "Deferred — exceeded question budget")

#### Scenario: Unresolved decision that was asked
- **GIVEN** a planning skill asks the user about an Unresolved decision
- **WHEN** the user cannot decide or defers
- **THEN** the Assumptions table row SHALL have Grade `Unresolved` and Rationale containing `Asked — {outcome description}`

#### Scenario: Unresolved decision deferred by agent
- **GIVEN** a planning skill identifies an Unresolved decision but has already used its question budget
- **WHEN** the agent records the decision without asking
- **THEN** the Assumptions table row SHALL have Grade `Unresolved` and Rationale containing `Deferred — exceeded question budget`

### Requirement: Updated Confidence Grades Output Visibility

The Confidence Grades table in `_context.md` SHALL be updated:

| Grade | Output Visibility (new) |
|-------|------------------------|
| **Certain** | Noted in Assumptions summary (was: "None — not worth mentioning") |
| **Confident** | Noted in Assumptions summary (unchanged) |
| **Tentative** | Noted in Assumptions summary, `/fab-clarify` suggested (unchanged) |
| **Unresolved** | Asked as question AND noted in Assumptions summary (was: "Asked as question (fab-new/continue), batched upfront (fab-ff/fab-fff)") |

#### Scenario: Certain decisions visible in summary
- **GIVEN** a planning skill generates an artifact with Certain assumptions
- **WHEN** the Assumptions summary is displayed
- **THEN** Certain decisions SHALL appear in the summary table
- **AND** the Assumptions table in the artifact SHALL include Certain rows

### Requirement: Updated Summary Line Format

The Assumptions summary line SHALL use the format:

```
{N} assumptions ({Ce} certain, {Co} confident, {T} tentative, {U} unresolved)
```

replacing the previous format `{N} assumptions ({C} confident, {T} tentative)`.

#### Scenario: Summary line with all grades
- **GIVEN** a spec with 7 Certain, 2 Confident, 1 Tentative, 0 Unresolved assumptions
- **WHEN** the summary line is generated
- **THEN** it SHALL read `10 assumptions (7 certain, 2 confident, 1 tentative, 0 unresolved)`

## Scoring Pipeline: calc-score.sh

### Requirement: Parse Only spec.md

`calc-score.sh` SHALL parse the `## Assumptions` table from `spec.md` only. It SHALL NOT read or parse `brief.md`.

The `brief_file` variable and `parse_assumptions "$brief_file"` call SHALL be removed.

#### Scenario: Score computed from spec only
- **GIVEN** a change with `brief.md` containing 10 assumptions and `spec.md` containing 8 assumptions
- **WHEN** `calc-score.sh` runs
- **THEN** it SHALL count grades from `spec.md` only (8 total)
- **AND** `brief.md` SHALL not be opened or parsed

#### Scenario: brief.md missing has no effect
- **GIVEN** a change directory with `spec.md` but no `brief.md`
- **WHEN** `calc-score.sh` runs
- **THEN** it SHALL complete successfully, scoring from `spec.md`

### Requirement: Remove Implicit Certain Carry-Forward

The block that reads previous Certain count from `.status.yaml` and computes implicit carry-forward (`implicit_certain = prev_certain - table_certain`) SHALL be removed.

With spec-only parsing, `total_certain` equals `table_certain` directly. There is no implicit/explicit distinction.

#### Scenario: Certain count comes directly from table
- **GIVEN** `spec.md` has 5 Certain rows in its Assumptions table
- **WHEN** `calc-score.sh` computes the confidence block
- **THEN** `certain` SHALL be `5` (not inflated by carry-forward from previous runs)

### Requirement: Fix AWK Column Index for Scores

The `parse_assumptions` AWK function SHALL extract the Scores column from `cols[6]`, not `cols[4]`.

In a 5-column markdown table (`| # | Grade | Decision | Rationale | Scores |`), splitting by `|` produces: `cols[1]=""`, `cols[2]="#"`, `cols[3]="Grade"`, `cols[4]="Decision"`, `cols[5]="Rationale"`, `cols[6]="Scores"`.

The current code reads `cols[4]` (Decision text) instead of `cols[6]` (Scores), causing all `S:\d+` pattern matches to fail silently.

#### Scenario: Dimension scores correctly parsed
- **GIVEN** an Assumptions table row `| 1 | Confident | Use JSON errors | Codebase signal | S:85 R:70 A:85 D:80 |`
- **WHEN** `parse_assumptions` processes this row
- **THEN** the output SHALL include `Confident|S:85 R:70 A:85 D:80`
- **AND** dimension aggregation SHALL use values S=85, R=70, A=85, D=80

#### Scenario: Old code would have failed
- **GIVEN** the same row as above
- **WHEN** the old `cols[4]` extraction runs
- **THEN** it would extract `Use JSON errors` (the Decision column) and grep for `S:\K[0-9]+` would find nothing
<!-- assumed: Documenting the bug for test clarity — confirmed by code inspection of cols indexing -->

### Requirement: Remove has_scores Detection

The `has_scores` variable and conditional logic in `parse_assumptions` SHALL be removed. Since the Scores column is now mandatory, `parse_assumptions` SHALL always extract `cols[6]` as the scores value.

The output format SHALL always be `grade|scores` (never bare `grade` without scores).

#### Scenario: No conditional scores detection
- **GIVEN** `parse_assumptions` processes an Assumptions table
- **WHEN** it encounters a data row
- **THEN** it SHALL extract `cols[3]` as grade and `cols[6]` as scores unconditionally
- **AND** the `has_scores` header check SHALL not exist in the code

### Requirement: Parse Unresolved Grade

The grade counting logic in `calc-score.sh` SHALL include an `unresolved` case in its grade matching. The hardcoded `unresolved=0` line SHALL be removed.

#### Scenario: Unresolved triggers score zero
- **GIVEN** `spec.md` contains an Assumptions table with 5 Certain, 2 Confident, and 1 Unresolved row
- **WHEN** `calc-score.sh` computes the score
- **THEN** `unresolved` SHALL be `1`
- **AND** the score SHALL be `0.0` (formula: `if unresolved > 0: score = 0.0`)

#### Scenario: No unresolved allows normal scoring
- **GIVEN** `spec.md` contains 5 Certain and 2 Confident rows with 0 Unresolved
- **WHEN** `calc-score.sh` computes the score
- **THEN** `unresolved` SHALL be `0`
- **AND** the score SHALL be `4.4` (`5.0 - 0.3*2 = 4.4`)

### Requirement: Graceful Transition for Existing Specs

During the transition period, `calc-score.sh` SHOULD handle existing specs that lack the Scores column gracefully. If a data row produces an empty or unparseable `cols[6]`, the dimension score aggregation for that row SHALL be skipped (row contributes to grade count but not to dimension means).

#### Scenario: Old-format spec without Scores column
- **GIVEN** an existing `spec.md` with a 4-column Assumptions table (no Scores column)
- **WHEN** `calc-score.sh` processes it
- **THEN** grade counting SHALL work correctly
- **AND** `cols[6]` will be empty, so no dimension scores are aggregated for those rows
- **AND** `has_fuzzy` SHALL remain false (no valid dimension data found)

### Requirement: Spec Generation Reads Brief Assumptions as Starting Point

`fab/.kit/skills/_generation.md` Spec Generation Procedure Step 6 SHALL specify that the spec-stage agent:

1. Reads `brief.md`'s `## Assumptions` table as its starting point
2. Confirms, upgrades, or overrides each brief assumption based on spec-level analysis
3. Adds new assumptions discovered during spec generation
4. Includes all four grades in the spec's own `## Assumptions` table

The spec's Assumptions table is an independent artifact — not a copy of the brief's table. It represents the spec-stage agent's own SRAD analysis, informed by but not constrained to the brief's assessments.

#### Scenario: Spec upgrades brief assumption
- **GIVEN** a brief has an assumption graded Tentative: "Use Redis for caching"
- **WHEN** the spec-stage agent finds `config.yaml` specifies Redis
- **THEN** the spec's Assumptions table SHALL include the same decision graded Certain (upgraded)
- **AND** the Rationale SHALL note the upgrade: "Confirmed by config — upgraded from brief Tentative"

#### Scenario: Spec overrides brief assumption
- **GIVEN** a brief has an assumption graded Confident: "Single cache layer"
- **WHEN** the spec-stage agent determines multi-layer is required by the architecture
- **THEN** the spec MAY override the decision with a different approach
- **AND** the Rationale SHALL note the override: "Overridden — spec analysis shows multi-layer required"

### Requirement: fab-ff Cumulative Summary — All Four Grades

`fab/.kit/skills/fab-ff.md` Step 1 (or equivalent summary tracking) SHALL track all four SRAD grades in the cumulative Assumptions summary across all generated stages.

## Design Decisions

1. **Spec-only scoring eliminates double-counting**
   - *Why*: Brief and spec represent the same decision space at different resolution levels. Counting both inflates grade totals. The spec is the authoritative decision record — the brief is a state-transfer mechanism.
   - *Rejected*: Deduplication by decision ID — fragile, requires agents to maintain stable IDs across stages.

2. **Scores column always mandatory (no detection logic)**
   - *Why*: Simplifies parsing (no conditional `has_scores` path), ensures every grade has a transparent rationale, and makes the output deterministic. The previous optional mode existed because fuzzy scoring was new; now it's the standard.
   - *Rejected*: Keeping optional with graceful fallback — maintains two code paths indefinitely.

3. **Brief assumptions as state-transfer, not scored**
   - *Why*: Pipeline stages execute in isolated agent contexts with no shared memory. The brief's Assumptions table is the only continuity mechanism between the brief-stage agent and the spec-stage agent. Including it in scoring would double-count decisions.
   - *Rejected*: Removing brief Assumptions entirely — loses the state-transfer function.
