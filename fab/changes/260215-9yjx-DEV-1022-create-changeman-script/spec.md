## Non-Goals

- Archive, list, or delete subcommands — scoped to `new` only; extend later without breaking changes
- Semantic slug validation (word choice, article removal) — AI judgment stays in the skill
- Changes to `stageman.sh`, `preflight.sh`, or `resolve-change.sh` — consumed as-is via CLI
- Changes to `status.yaml` template — consumed as-is via `sed`

## Scripts: changeman.sh — Change Creation Script

### Requirement: Script Conventions

`changeman.sh` SHALL be located at `fab/.kit/scripts/lib/changeman.sh` and follow existing lib script conventions:

- Shebang: `#!/usr/bin/env bash`
- `set -euo pipefail`
- Path resolution via `readlink -f "$0"` to support symlinks (matches stageman.sh pattern)
- Derive `FAB_ROOT` (the `fab/` directory) from script location: `lib/ → scripts/ → .kit/ → fab/`
- `STAGEMAN` variable pointing to `lib/stageman.sh` for CLI subprocess calls
- yq is NOT required (uses `sed` for template filling)

#### Scenario: Path resolution from standard location
- **GIVEN** changeman.sh lives at `fab/.kit/scripts/lib/changeman.sh`
- **WHEN** invoked from any working directory
- **THEN** FAB_ROOT resolves to the `fab/` directory
- **AND** STAGEMAN resolves to `fab/.kit/scripts/lib/stageman.sh`

#### Scenario: Path resolution from symlink
- **GIVEN** changeman.sh is symlinked from `src/lib/changeman/changeman.sh`
- **WHEN** invoked via the symlink
- **THEN** paths resolve through the symlink to the real `fab/.kit/scripts/lib/` location

### Requirement: `new` Subcommand Interface

`changeman.sh` SHALL support the following interface:

```
changeman.sh new --slug <slug> [--change-id <4char>] [--log-args <description>]
changeman.sh --help
```

| Flag | Required | Description |
|------|----------|-------------|
| `--slug <slug>` | Yes | The suffix portion of the folder name (e.g., `add-oauth` or `DEV-988-add-oauth`). Generated by the calling skill. |
| `--change-id <4char>` | No | Explicit 4-character alphanumeric ID (e.g., from backlog). If omitted, a random ID is generated. |
| `--log-args <description>` | No | Description string passed to `stageman.sh log-command` as the args parameter. |

The `new` subcommand SHALL perform these operations in sequence:
1. Parse and validate arguments
2. Generate date prefix (YYMMDD)
3. Generate or validate change ID
4. Validate slug format
5. Construct folder name
6. Collision detection
7. Create directory
8. Detect `created_by`
9. Initialize `.status.yaml` from template
10. Call `stageman.sh set-state`
11. Call `stageman.sh log-command` (if `--log-args` provided)
12. Print folder name to stdout

#### Scenario: Minimal invocation (slug only)
- **GIVEN** `fab/changes/` directory exists
- **WHEN** `changeman.sh new --slug add-oauth` is invoked
- **THEN** a random 4-char alphanumeric ID is generated
- **AND** the folder `fab/changes/{YYMMDD}-{XXXX}-add-oauth/` is created
- **AND** `.status.yaml` is initialized with `name: {YYMMDD}-{XXXX}-add-oauth`
- **AND** the folder name (not full path) is printed to stdout

#### Scenario: Full invocation with all flags
- **GIVEN** `fab/changes/` directory exists
- **WHEN** `changeman.sh new --slug DEV-988-add-oauth --change-id a7k2 --log-args "Add OAuth support"` is invoked
- **THEN** folder `fab/changes/260215-a7k2-DEV-988-add-oauth/` is created
- **AND** `stageman.sh log-command` is called with args `"Add OAuth support"`
- **AND** the folder name is printed to stdout

#### Scenario: Missing required --slug flag
- **GIVEN** no `--slug` flag is provided
- **WHEN** `changeman.sh new` is invoked
- **THEN** an error message is printed to stderr
- **AND** exit code is non-zero

#### Scenario: --help
- **GIVEN** the user invokes `changeman.sh --help`
- **WHEN** the command runs
- **THEN** usage information is printed to stdout
- **AND** exit code is 0

### Requirement: Date Generation

The date prefix SHALL be generated as a 6-digit `YYMMDD` string from today's date using `date +%y%m%d`.

#### Scenario: Date format
- **GIVEN** today is 2026-02-15
- **WHEN** the date prefix is generated
- **THEN** the result is `260215`

### Requirement: Change ID Generation

When `--change-id` is NOT provided, `changeman.sh` SHALL generate a random 4-character string from the set `[a-z0-9]` using `/dev/urandom`.

When `--change-id` IS provided, the value SHALL be validated: it MUST be exactly 4 characters matching `^[a-z0-9]{4}$`. Invalid values SHALL produce an error.

#### Scenario: Random ID generation
- **GIVEN** `--change-id` is not provided
- **WHEN** the ID is generated
- **THEN** the result is 4 characters from `[a-z0-9]`
- **AND** the generation uses `/dev/urandom` (e.g., `LC_ALL=C tr -dc 'a-z0-9' < /dev/urandom | head -c4`)

#### Scenario: Valid provided ID
- **GIVEN** `--change-id a7k2` is provided
- **WHEN** the ID is validated
- **THEN** `a7k2` is accepted and used as-is

#### Scenario: Invalid provided ID (too long)
- **GIVEN** `--change-id abcde` is provided
- **WHEN** the ID is validated
- **THEN** an error is printed to stderr: `"ERROR: Invalid change-id 'abcde' (expected 4 lowercase alphanumeric chars)"`
- **AND** exit code is non-zero

#### Scenario: Invalid provided ID (uppercase)
- **GIVEN** `--change-id A7K2` is provided
- **WHEN** the ID is validated
- **THEN** an error is printed to stderr
- **AND** exit code is non-zero

### Requirement: Slug Validation

The `--slug` value SHALL be validated with light structural checks only:

1. Non-empty
2. Matches pattern: `^[a-zA-Z0-9][a-zA-Z0-9-]*$` (alphanumeric start, alphanumeric/hyphen body). Uppercase is allowed to support Linear issue IDs (e.g., `DEV-988-add-oauth`).
3. No trailing hyphen

The script SHALL NOT perform semantic validation (word count, article detection, abbreviation checks) — that is the calling skill's responsibility.

#### Scenario: Valid slug without issue ID
- **GIVEN** slug is `add-oauth-support`
- **WHEN** validated
- **THEN** validation passes

#### Scenario: Valid slug with issue ID prefix
- **GIVEN** slug is `DEV-988-add-oauth`
- **WHEN** validated
- **THEN** validation passes (uppercase allowed for issue IDs)

#### Scenario: Empty slug
- **GIVEN** slug is an empty string
- **WHEN** validated
- **THEN** error: `"ERROR: --slug must not be empty"`

#### Scenario: Slug with invalid characters
- **GIVEN** slug is `add_oauth!`
- **WHEN** validated
- **THEN** error: `"ERROR: Invalid slug format (expected alphanumeric and hyphens)"`

### Requirement: Folder Name Construction

The folder name SHALL be constructed as: `{YYMMDD}-{XXXX}-{slug}` where:
- `{YYMMDD}` is the generated date prefix
- `{XXXX}` is the change ID (random or provided)
- `{slug}` is the validated slug value (may include issue ID prefix)

All components are joined with hyphens. The slug is used as-is (the skill is responsible for formatting including lowercase conversion of the slug portion).

#### Scenario: Folder name with slug only
- **GIVEN** date is `260215`, ID is `a7k2`, slug is `add-oauth`
- **WHEN** folder name is constructed
- **THEN** result is `260215-a7k2-add-oauth`

#### Scenario: Folder name with issue ID in slug
- **GIVEN** date is `260215`, ID is `a7k2`, slug is `DEV-988-add-oauth`
- **WHEN** folder name is constructed
- **THEN** result is `260215-a7k2-DEV-988-add-oauth`

### Requirement: Collision Detection

Before creating the directory, `changeman.sh` SHALL check for an existing folder with the same name under `fab/changes/`.

**Provided ID collision** (via `--change-id`): The script SHALL exit with a fatal error and non-zero exit code. The error message SHALL include the colliding folder name.

**Random ID collision**: The script SHALL regenerate the random ID and retry, up to a maximum of 10 attempts. If all 10 attempts collide, exit with a fatal error.

#### Scenario: Provided ID collides
- **GIVEN** `--change-id 9yjx` is provided
- **AND** `fab/changes/260215-9yjx-some-change/` already exists
- **WHEN** collision is detected
- **THEN** error: `"ERROR: Change ID '9yjx' already in use (260215-9yjx-some-change)"`
- **AND** exit code is non-zero (no retry)

#### Scenario: Random ID collides, retry succeeds
- **GIVEN** random ID generation first produces an ID that collides
- **WHEN** the collision is detected
- **THEN** a new random ID is generated
- **AND** folder creation proceeds with the new ID

#### Scenario: Random ID exhausts retries
- **GIVEN** all 10 random ID attempts collide
- **WHEN** the retry limit is reached
- **THEN** error: `"ERROR: Failed to generate unique change ID after 10 attempts"`
- **AND** exit code is non-zero

### Requirement: Directory Creation

The directory SHALL be created with `mkdir` (not `mkdir -p`). The parent directory `fab/changes/` is guaranteed to exist by `init-scaffold.sh`. Using plain `mkdir` provides a natural race-condition guard — if two concurrent invocations generate the same folder name, the second `mkdir` fails.

#### Scenario: Successful directory creation
- **GIVEN** `fab/changes/` exists and the folder name is unique
- **WHEN** `mkdir fab/changes/{name}` is executed
- **THEN** the directory is created successfully

#### Scenario: Parent directory missing
- **GIVEN** `fab/changes/` does not exist
- **WHEN** `mkdir fab/changes/{name}` is executed
- **THEN** `mkdir` fails naturally
- **AND** the error propagates (script exits non-zero due to `set -e`)

### Requirement: `created_by` Detection

The `created_by` field SHALL be determined using a fallback chain:

1. `gh api user --jq .login` (GitHub username — primary)
2. `git config user.name` (local git config — fallback)
3. `"unknown"` (final fallback)

Each step in the chain SHALL fail silently (no warning, no error output). The first successful non-empty result is used.

#### Scenario: GitHub API available
- **GIVEN** `gh` CLI is authenticated
- **WHEN** `gh api user --jq .login` returns `"sahil-weaver"`
- **THEN** `created_by` is set to `sahil-weaver`

#### Scenario: GitHub API fails, git config available
- **GIVEN** `gh api user` fails (not authenticated or not installed)
- **AND** `git config user.name` returns `"Sahil"`
- **WHEN** `created_by` is detected
- **THEN** `created_by` is set to `Sahil`
- **AND** no error or warning is printed

#### Scenario: Both fail
- **GIVEN** both `gh api user` and `git config user.name` fail
- **WHEN** `created_by` is detected
- **THEN** `created_by` is set to `unknown`

### Requirement: `.status.yaml` Initialization

The `.status.yaml` file SHALL be created by reading `fab/.kit/templates/status.yaml` and performing `sed` substitution on placeholder markers:

| Placeholder | Replacement |
|-------------|-------------|
| `{NAME}` | The constructed folder name |
| `{CREATED}` | ISO 8601 timestamp with timezone (`date -Iseconds`) |
| `{CREATED_BY}` | The detected `created_by` value |

The `sed` substitution SHALL be a single pipeline replacing all three placeholders. The result SHALL be written to `fab/changes/{name}/.status.yaml`.

#### Scenario: Template filling
- **GIVEN** the template contains `name: {NAME}`, `created: {CREATED}`, `created_by: {CREATED_BY}`
- **WHEN** sed substitution runs with name=`260215-a7k2-add-oauth`, created=`2026-02-15T14:30:00+05:30`, created_by=`sahil-weaver`
- **THEN** the output file contains `name: 260215-a7k2-add-oauth`, `created: 2026-02-15T14:30:00+05:30`, `created_by: sahil-weaver`
- **AND** all other template content (progress, checklist, confidence, stage_metrics) is preserved as-is

### Requirement: Stageman Integration

After creating the directory and initializing `.status.yaml`, `changeman.sh` SHALL invoke `stageman.sh` via CLI subprocess calls (never `source`):

1. `$STAGEMAN set-state <status_file> intake active fab-new` — activate the intake stage with driver `fab-new`
2. `$STAGEMAN log-command <change_dir> "fab-new" "<log_args>"` — only if `--log-args` was provided

Both calls SHALL use the absolute path to `stageman.sh` resolved during path setup. If either call fails, the script SHALL exit non-zero (inheriting `set -e` behavior). The driver is hardcoded as `"fab-new"` since that is the sole consumer.

#### Scenario: Stageman calls succeed
- **GIVEN** directory and `.status.yaml` are created
- **WHEN** stageman calls are made
- **THEN** `.status.yaml` reflects `intake: active` with `stage_metrics.intake` populated
- **AND** `.history.jsonl` contains the log entry (if `--log-args` provided)

#### Scenario: Stageman call fails
- **GIVEN** `stageman.sh` exits non-zero (e.g., invalid state)
- **WHEN** the failure occurs
- **THEN** the script exits non-zero
- **AND** the partially-created change directory remains on disk (no cleanup — idempotent retry is preferred)

### Requirement: Output Contract

**stdout**: On success, the script SHALL print exactly one line: the folder name (not the full path). Example: `260215-a7k2-add-oauth`

**stderr**: All errors and diagnostic messages go to stderr. Error messages SHALL be prefixed with `ERROR:`.

**Exit codes**: 0 on success, non-zero on any failure.

#### Scenario: Stdout contains only folder name
- **GIVEN** a successful `changeman.sh new --slug add-oauth` invocation
- **WHEN** stdout is captured
- **THEN** it contains exactly `260215-{XXXX}-add-oauth` (one line, no trailing newline beyond the standard)

## Skills: fab-new Skill Refactoring

### Requirement: Step Consolidation

The refactored `fab-new.md` SHALL collapse Steps 1 (Generate Folder Name → only slug generation), 3 (Create Change Directory), and 4 (Initialize `.status.yaml`) into a single `changeman.sh new` invocation. The skill's responsibility narrows to:

1. **Step 0**: Parse Input (unchanged)
2. **Step 1**: Generate Slug — AI-driven word selection, article removal, issue ID prefixing. Produces the `--slug` value. If a backlog ID was detected, it becomes the `--change-id` value.
3. **Step 2**: Gap Analysis (unchanged)
4. **Step 3**: Create Change — single `changeman.sh new` call with appropriate flags. Capture folder name from stdout.
5. **Step 4**: Generate `intake.md` (unchanged, uses captured folder name)
6. **Step 5**: SRAD-Based Question Selection (unchanged)
7. **Step 6**: Activate Change (unchanged, conditional)

This reduces the step count from 7 to 7 (renumbered) but consolidates 3 bash operations (mkdir, sed template fill, created_by detection) plus 2 stageman calls into one permission prompt.

#### Scenario: Successful creation via changeman
- **GIVEN** the skill has generated slug `add-oauth` and backlog ID `a7k2`
- **WHEN** Step 3 executes
- **THEN** `changeman.sh new --slug add-oauth --change-id a7k2 --log-args "Add OAuth support"` is called
- **AND** the folder name from stdout is used for all subsequent steps

#### Scenario: Successful creation with Linear issue
- **GIVEN** the skill parsed Linear issue `DEV-988` and generated slug `DEV-988-add-oauth`
- **WHEN** Step 3 executes
- **THEN** `changeman.sh new --slug DEV-988-add-oauth --log-args "DEV-988: Add OAuth support"` is called

### Requirement: Error Table Simplification

The error handling table in `fab-new.md` SHALL be simplified. The previous entries for "Backlog ID collision", "Random ID collision", and any directory/template creation errors SHALL be consolidated into a single entry:

| Condition | Action |
|-----------|--------|
| `changeman.sh` failure | Surface stderr output to user and stop |

The changeman.sh script owns collision detection, directory creation, template filling, and stageman integration errors. The skill surfaces whatever changeman reports without duplicating error logic.

#### Scenario: changeman.sh reports collision
- **GIVEN** `changeman.sh new` exits non-zero with stderr `"ERROR: Change ID '9yjx' already in use"`
- **WHEN** the skill processes the failure
- **THEN** the error message is surfaced to the user
- **AND** the skill stops (does not attempt recovery — user redirected to existing change)

## Design Decisions

1. **Slug includes issue ID prefix**: The `--slug` value passed to `changeman.sh` includes the optional issue ID (e.g., `DEV-988-add-oauth`), not just the word slug. This keeps changeman.sh unaware of Linear/issue concepts — it simply concatenates `{YYMMDD}-{XXXX}-{slug}`. The skill is responsible for constructing the appropriate slug.
   - *Why*: Minimizes changeman.sh's concerns. The naming format `{YYMMDD}-{XXXX}-[{ISSUE}-]{slug}` from config.yaml is a skill-level convention, not a script-level one.
   - *Rejected*: Separate `--issue` flag — adds complexity to changeman.sh for no benefit since the skill already knows how to compose the slug.

2. **No cleanup on partial failure**: If `changeman.sh` creates the directory but a subsequent stageman call fails, the directory is left on disk. Re-running the skill will hit a collision (for provided IDs) or succeed with a new random ID.
   - *Why*: Cleanup logic adds complexity and can mask root causes. The user can `rm -rf` the partial directory if needed. This matches the "idempotent retry" philosophy.
   - *Rejected*: `trap` cleanup — risks deleting intentional directories in edge cases.
