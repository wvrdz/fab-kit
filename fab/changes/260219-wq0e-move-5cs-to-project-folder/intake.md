# Intake: Move 5 Cs and VERSION into fab/project/

**Change**: 260219-wq0e-move-5cs-to-project-folder
**Created**: 2026-02-19
**Status**: Draft

## Origin

> Move the 5 Cs (constitution.md, context.md, code-quality.md, code-review.md, config.yaml) and VERSION file into a new fab/project/ subfolder. Update all references across the codebase — scripts, skills, agents, specs, README, and any path constants — to point to the new locations.

User-initiated during a discussion about folder naming. The 5 Cs currently live flat in `fab/` alongside unrelated files (`backlog.md`, `current`, `.kit/`, `changes/`). Moving them into `fab/project/` creates a clean top-level hierarchy:

```
fab/
├── .kit/           # the tool (engine)
├── project/        # project identity (5 Cs + VERSION)
├── changes/        # active work
└── backlog.md      # idea queue
```

## Why

1. **Cluttered `fab/` root** — The 5 Cs, VERSION, backlog.md, current pointer, changes/, and .kit/ all live at the same level. There's no visual separation between "project identity files" and "workflow machinery."

2. **If we don't fix it** — As more project-level files are added (or the kit evolves), the `fab/` directory becomes harder to navigate. The 5 Cs are conceptually grouped (they're all project-level configuration that shapes AI behavior) but physically scattered among unrelated items.

3. **Why `fab/project/`** — Discussed alternatives: `guide/`, `profile/`, `rails/`, `config/`. `project/` won best because it honestly describes the contents (project-level definitions), is short, and creates a clean triad with `.kit/` (engine) and `changes/` (work). `config/` was rejected because `config/config.yaml` is redundant.
