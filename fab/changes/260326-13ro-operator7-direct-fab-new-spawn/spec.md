## Operator: Raw Text Spawn Path

### Requirement: Direct fab-new Invocation for Raw Text

When the operator receives a raw text description (not a backlog ID or Linear issue), it SHALL pass the description directly to `/fab-new` as part of the spawn command. The operator SHALL NOT create a backlog entry via `idea add` as an intermediate step.

The spawn sequence for raw text SHALL be:
1. Create worktree (`wt create --non-interactive`)
2. Resolve dependencies (cherry-pick `depends_on` entries if any)
3. Spawn agent: `tmux new-window -n "fab-<wt>" -c <worktree-path> "<spawn_cmd> '/fab-new <description>'"` — where `<description>` is the raw text as provided
4. Enroll in monitored set
5. On completion: merge PR, optionally archive

#### Scenario: Raw text spawn without idea add
- **GIVEN** the operator receives a raw text request like "fix login after password reset"
- **WHEN** the operator spawns an agent for this request
- **THEN** the operator creates a worktree and spawns an agent with `/fab-new fix login after password reset` as the command
- **AND** no `idea add` command is executed
- **AND** no backlog entry is created in `fab/backlog.md`

#### Scenario: Raw text with special characters
- **GIVEN** the operator receives a raw text request containing quotes or special characters
- **WHEN** the operator constructs the spawn command
- **THEN** the description is properly escaped in the tmux send-keys or new-window command

### Requirement: Updated Explanatory Paragraph

The explanatory paragraph following the "From raw text" section SHALL be updated to remove the `idea add` reference. The paragraph SHALL explain that `/fab-new` handles traceability directly via its intake Origin section.

#### Scenario: Paragraph content after edit
- **GIVEN** the "From raw text" section has been updated
- **WHEN** a reader reviews the explanatory paragraph
- **THEN** the paragraph explains that `/fab-new` creates the intake with traceability (Origin section captures the raw input)
- **AND** no reference to `idea add` exists in the paragraph

### Requirement: Backlog ID and Linear Issue Paths Unchanged

The "From backlog ID or Linear issue" path SHALL remain unchanged. It SHALL continue to use `idea show <id>` for backlog lookups and Linear issue resolution as before.

#### Scenario: Backlog ID spawn is unaffected
- **GIVEN** the operator receives a backlog ID like `90g5`
- **WHEN** the operator processes the request
- **THEN** the operator follows the existing structured flow (idea show → worktree → spawn with `/fab-new <id>`)
