# Spec: Fix wt open "Open Here" cd Mechanism

**Change**: 260328-3tds-fix-wt-open-here-cd
**Created**: 2026-03-28
**Affected memory**: `docs/memory/fab-workflow/distribution.md`

## Non-Goals

- Fish shell support — deferred to future enhancement
- Auto-detecting and patching the user's shell profile — user must opt-in manually
- Changing the existing `cd -- "/path"` stdout protocol — it remains the core mechanism

## Packages: Shell Integration

### Requirement: `wt shell-setup` Subcommand

`wt` SHALL provide a `shell-setup` subcommand that outputs a shell wrapper function to stdout. The function, when sourced into the user's shell, intercepts `wt` invocations and `eval`s any `cd` output from the binary.

The subcommand SHALL detect the current shell from the `SHELL` environment variable (basename extraction) and output the appropriate wrapper. Supported shells: `bash` and `zsh`. For unrecognized shells, the subcommand SHALL print an error to stderr and exit 1.

The wrapper function SHALL:
1. Set `WT_WRAPPER=1` in the environment before invoking `command wt`
2. Capture all stdout line-by-line, printing each line
3. Track the last line of output
4. After the binary exits, check if the last line starts with `cd ` (literal prefix match)
5. If yes, `eval` the last line in the current shell context
6. Return the binary's exit code

#### Scenario: User sets up shell integration (bash)
- **GIVEN** the user's `SHELL` is `/bin/bash`
- **WHEN** `eval "$(wt shell-setup)"` is added to `~/.bashrc` and a new shell is opened
- **THEN** the `wt` shell function is defined in the session
- **AND** running `wt open` and selecting "Open here" changes the shell's working directory

#### Scenario: User sets up shell integration (zsh)
- **GIVEN** the user's `SHELL` is `/bin/zsh`
- **WHEN** `eval "$(wt shell-setup)"` is added to `~/.zshrc` and a new shell is opened
- **THEN** the `wt` shell function is defined in the session
- **AND** behavior is identical to bash

#### Scenario: Unsupported shell
- **GIVEN** the user's `SHELL` is `/usr/bin/fish`
- **WHEN** `wt shell-setup` is run
- **THEN** stderr shows `wt shell-setup: unsupported shell "fish" (supported: bash, zsh)`
- **AND** exit code is 1

#### Scenario: Shell function preserves normal wt output
- **GIVEN** the shell wrapper is active
- **WHEN** `wt list` is run (no `cd` in output)
- **THEN** all output is printed normally to stdout
- **AND** no `eval` occurs

#### Scenario: Shell function handles cd with special characters in path
- **GIVEN** the shell wrapper is active
- **WHEN** "Open here" is selected for a worktree at `/home/user/my projects/repo.worktrees/swift-fox`
- **THEN** the `cd -- "/home/user/my projects/repo.worktrees/swift-fox"` line is eval'd
- **AND** the shell CWD changes to that path

### Requirement: WT_WRAPPER Environment Variable

The shell wrapper function generated by `wt shell-setup` SHALL set `WT_WRAPPER=1` in the environment before invoking `command wt "$@"`. This allows the Go binary to detect whether it is running through the wrapper.

#### Scenario: Wrapper sets environment variable
- **GIVEN** the shell wrapper is active
- **WHEN** any `wt` command is run
- **THEN** the Go binary receives `WT_WRAPPER=1` in its environment

### Requirement: Stderr Hint When Wrapper Not Detected

When the `open_here` app is selected and `WT_WRAPPER` is not set (or empty), the Go binary SHALL print a one-line hint to stderr after printing the `cd` command to stdout:

```
hint: run "wt shell-setup" for setup instructions
```

When `WT_WRAPPER=1` is set, the hint SHALL be suppressed.

#### Scenario: Open here without wrapper — hint shown
- **GIVEN** the shell wrapper is NOT active (`WT_WRAPPER` unset)
- **WHEN** "Open here" is selected from the `wt open` menu
- **THEN** stdout contains `cd -- "/path/to/worktree"`
- **AND** stderr contains `hint: run "wt shell-setup" for setup instructions`

#### Scenario: Open here with wrapper — hint suppressed
- **GIVEN** the shell wrapper IS active (`WT_WRAPPER=1`)
- **WHEN** "Open here" is selected from the `wt open` menu
- **THEN** stdout contains `cd -- "/path/to/worktree"`
- **AND** stderr is empty

#### Scenario: Open here via wt create without wrapper
- **GIVEN** the shell wrapper is NOT active
- **WHEN** "Open here" is selected during `wt create`
- **THEN** the same hint is printed to stderr
- **AND** the `cd` command is printed to stdout (path suppression in create still works)

### Requirement: Help Text Update

The `wt --help` long description SHALL be updated to reference `wt shell-setup` instead of showing the raw wrapper function. The new text SHALL direct users to run `eval "$(wt shell-setup)"` in their shell profile.

#### Scenario: Help text shows setup command
- **GIVEN** a user runs `wt --help`
- **WHEN** the help text is displayed
- **THEN** it includes `eval "$(wt shell-setup)"` as the recommended setup
- **AND** the raw wrapper function code is no longer shown

## Design Decisions

1. **`shell-setup` over `init-shell`**: The `init` subcommand already exists (runs worktree init scripts). Using `shell-setup` avoids confusion and is more descriptive of what the command does.
   - *Why*: `wt init` is taken; `shell-setup` clearly communicates "set up shell integration"
   - *Rejected*: `init-shell` (too close to existing `init` command, could confuse tab-completion)

2. **Identical wrapper for bash and zsh**: Both shells support the same syntax (`local`, `read -r`, `< <(...)`, `eval`). A single wrapper definition works for both, simplifying the implementation.
   - *Why*: bash and zsh share sufficient syntax for this wrapper
   - *Rejected*: separate per-shell implementations (unnecessary complexity)

3. **Hint on stderr, not stdout**: The hint goes to stderr so it doesn't interfere with the `cd` protocol on stdout. Scripts piping `wt` output won't see the hint.
   - *Why*: stdout is reserved for the cd protocol; stderr is the standard channel for notices
   - *Rejected*: stdout hint (would break the wrapper's last-line detection)
