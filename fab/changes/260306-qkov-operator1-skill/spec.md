# Spec: Add `/fab-operator1` Skill

**Change**: 260306-qkov-operator1-skill
**Created**: 2026-03-07
**Affected memory**: `docs/memory/fab-workflow/execution-skills.md`, `docs/memory/fab-workflow/kit-architecture.md`, `docs/memory/fab-workflow/distribution.md`

## Non-Goals

- Autopilot / sequential pipeline execution — deferred to a separate change. The operator does not autonomously drive changes through the pipeline.
- Standing orders / persistent instructions — session context suffices for v1. Future home: `fab/project/operator1.yaml`.
- Multi-repo coordination — fab-kit's worktree model is single-repo only.
- Background polling / daemon behavior — the operator is user-driven, not event-driven.
- Loading change-specific artifacts (intakes, specs, tasks) — the operator only needs coordination primitives.

## Skill: `/fab-operator1`

### Requirement: Skill File Structure

The skill SHALL be implemented as a markdown skill file at `fab/.kit/skills/fab-operator1.md`. It SHALL use standard skill frontmatter (name, description). The deployed copy SHALL be generated by `fab-sync.sh` to `.claude/skills/fab-operator1/`.

#### Scenario: Skill file exists and is deployable
- **GIVEN** `fab/.kit/skills/fab-operator1.md` exists with valid frontmatter
- **WHEN** `fab-sync.sh` runs
- **THEN** a deployed copy is created at `.claude/skills/fab-operator1/`

### Requirement: Context Loading

The skill SHALL load the always-load layer (`_preamble.md` §1) — the same 7 files as `/fab-discuss`. It SHALL NOT run preflight. It SHALL NOT load change-specific artifacts (intakes, specs, tasks). The operator's context window is reserved for coordination state, not change content.

#### Scenario: Operator loads only coordination context
- **GIVEN** the operator skill is invoked
- **WHEN** context loading begins
- **THEN** it reads config.yaml, constitution.md, context.md (optional), code-quality.md (optional), code-review.md (optional), `docs/memory/index.md`, and `docs/specs/index.md`
- **AND** it does NOT read any `fab/changes/*/intake.md`, `spec.md`, or `tasks.md`

### Requirement: Orientation on Start

On invocation, the operator SHALL display the pane map (via `fab pane-map`) and a summary of all changes (via `fab status show --all`), then signal readiness for coordination commands.

#### Scenario: Operator startup with active agents
- **GIVEN** 3 tmux panes are running fab agents
- **WHEN** the operator skill is invoked
- **THEN** it displays the pane map table (Pane, Worktree, Change, Stage, Agent columns)
- **AND** displays a ready signal: "Ready for coordination commands."

#### Scenario: Operator startup outside tmux
- **GIVEN** the `$TMUX` environment variable is unset
- **WHEN** the operator skill is invoked
- **THEN** it displays: "Warning: not inside a tmux session. Pane map and send-keys unavailable. Status-only mode."
- **AND** it can still run `fab status show --all` for status queries

### Requirement: State Re-derivation

The operator MUST re-query live state (`fab pane-map`, `fab status show --all`, `fab runtime`) before every action. It SHALL NOT rely on stale values from conversation memory.

#### Scenario: Pane dies between actions
- **GIVEN** the operator displayed a pane map showing pane %3 with change r3m7
- **WHEN** the user says "send /fab-continue to r3m7"
- **THEN** the operator refreshes the pane map before sending
- **AND** if pane %3 no longer exists, reports: "Pane for r3m7 is gone (agent exited or tab closed)."

### Requirement: Seven Use Cases

The operator SHALL support seven use cases. Each use case follows the pattern: interpret user intent → refresh state → validate preconditions → execute → report.

#### Scenario: UC1 — Broadcast command to all idle agents
- **GIVEN** 3 agents exist, 2 are idle and 1 is active
- **WHEN** user says "run /fab-continue in all idle agents"
- **THEN** the operator refreshes pane map, filters for idle agents
- **AND** announces: "Sending /fab-continue to {change1} (%N), {change2} (%M)"
- **AND** sends `/fab-continue` to each idle agent via `fab send-keys`
- **AND** reports: "Done. 2 agents received /fab-continue."

#### Scenario: UC2 — Sequenced rebase after completion
- **GIVEN** agent r3m7 is at the apply stage and agent k8ds is idle
- **WHEN** user says "when r3m7 finishes, rebase k8ds on main"
- **THEN** the operator holds the instruction in conversation context
- **AND** on the next user interaction (or when explicitly asked to check), queries r3m7's status
- **AND** when r3m7 reaches hydrate or later, sends rebase command to k8ds via `fab send-keys`

#### Scenario: UC3 — Merge completed PRs
- **GIVEN** 2 changes are at `ship` stage with PRs created
- **WHEN** user says "merge all PRs that are ready"
- **THEN** the operator identifies changes with PRs via `fab status get-prs`
- **AND** confirms before executing (destructive action): "Will merge PRs for {change1}, {change2}. Proceed?"
- **AND** on confirmation, runs `gh pr merge` from the operator's own shell

#### Scenario: UC4 — Spawn new worktree + agent from idea
- **GIVEN** `fab/backlog.md` contains an idea about "retry logic"
- **WHEN** user says "start working on the retry logic idea"
- **THEN** the operator looks up the idea via `idea show "retry logic"`
- **AND** creates a worktree via `wt-create --non-interactive`
- **AND** opens a new tmux tab in that worktree
- **AND** sends `/fab-new <description>` to the new agent

#### Scenario: UC5 — Status dashboard
- **GIVEN** multiple agents are running
- **WHEN** user says "what's everyone doing?"
- **THEN** the operator refreshes the pane map and `fab status show --all`
- **AND** presents a concise human-readable summary with change name, stage, and agent state

#### Scenario: UC6 — Unstick a stuck agent
- **GIVEN** agent r3m7 has been idle at the apply stage
- **WHEN** user says "r3m7 looks stuck, nudge it"
- **THEN** the operator confirms r3m7 is idle via `fab runtime`
- **AND** announces: "Sending /fab-continue to r3m7 (%N)"
- **AND** sends `/fab-continue` via `fab send-keys`
- **AND** reports the send
- **AND** if a second nudge is requested for the same agent, warns: "Already nudged r3m7 once. Manual investigation recommended."

#### Scenario: UC7 — Notification surface
- **GIVEN** user says "tell me when r3m7 finishes"
- **WHEN** the user next interacts with the operator (or asks "any updates?")
- **THEN** the operator checks r3m7's status
- **AND** if r3m7 has reached hydrate or later, reports: "r3m7 finished — now at {stage}."
- **AND** if not finished, reports: "r3m7 still at {stage}."

### Requirement: Confirmation Model

Actions SHALL be categorized into three risk tiers with corresponding confirmation behavior:

| Risk | Examples | Behavior |
|------|----------|----------|
| Read-only | Status check, pane map | No confirmation |
| Recoverable | Send `/fab-continue`, rebase | Announce before sending |
| Destructive | Merge PR, archive, delete worktree | Confirm before executing |

#### Scenario: Destructive action requires confirmation
- **GIVEN** user says "merge all ready PRs"
- **WHEN** the operator identifies PRs to merge
- **THEN** it lists the PRs and asks: "Proceed?"
- **AND** only executes after user confirms

#### Scenario: Recoverable action announced but not confirmed
- **GIVEN** user says "send /fab-continue to all idle agents"
- **WHEN** the operator identifies idle agents
- **THEN** it announces what it will send and to whom
- **AND** sends without waiting for confirmation

### Requirement: Pre-Send Validation

Before sending keys to any pane via `fab send-keys`, the operator MUST:
1. Verify the target pane still exists (via refreshed pane map)
2. Check the agent is idle via `fab runtime`
3. If the agent is not idle, warn the user and ask for confirmation before sending

#### Scenario: Send to busy agent blocked
- **GIVEN** agent k8ds is active (not idle)
- **WHEN** user says "send /fab-continue to k8ds"
- **THEN** the operator warns: "k8ds is currently active. Sending may corrupt its work. Send anyway?"
- **AND** only sends if user confirms

#### Scenario: Target pane gone
- **GIVEN** pane for r3m7 no longer exists
- **WHEN** user says "send /fab-continue to r3m7"
- **THEN** the operator reports: "Pane for r3m7 is gone (agent exited or tab closed)."
- **AND** does NOT attempt to send

### Requirement: Bounded Retries and Escalation

Every automatic action SHALL have a bounded retry count. When the budget is exhausted, the operator escalates to the user.

| Situation | Max retries | Escalation |
|-----------|-------------|------------|
| Stuck agent nudge | 1 | "r3m7 appears stuck at {stage}. Manual investigation recommended." |
| Rebase conflict | 0 (never auto-resolve) | Immediately flag to user |

#### Scenario: Nudge limit reached
- **GIVEN** the operator already nudged r3m7 once
- **WHEN** user asks to nudge r3m7 again
- **THEN** the operator warns: "Already nudged r3m7 once. Manual investigation recommended."
- **AND** sends only if user explicitly insists

### Requirement: Context Discipline

The operator SHALL NOT load change artifacts (intakes, specs, tasks). It SHALL report concisely. It SHALL delegate diagnosis to the user. For terminal output inspection, it MAY use `tmux capture-pane -t <pane> -p` to read recent output but SHALL NOT deeply analyze codebase work.

#### Scenario: User asks what went wrong with an agent
- **GIVEN** user says "what's r3m7's last output?"
- **WHEN** the operator processes the request
- **THEN** it runs `tmux capture-pane -t <pane> -p` to read recent terminal output
- **AND** presents the output to the user
- **AND** does NOT attempt to load r3m7's spec, tasks, or source files

### Requirement: Not a Lifecycle Enforcer

The operator SHALL NOT validate stage transitions, enforce pipeline rules, or manage agent lifecycle. Individual agents self-govern via their own pipeline skills. The operator handles only cross-agent coordination.

#### Scenario: Agent at unexpected stage
- **GIVEN** agent r3m7 is at `intake` stage
- **WHEN** the user asks the operator to check status
- **THEN** the operator reports the stage factually
- **AND** does NOT attempt to advance or correct the agent's pipeline

## CLI: `fab send-keys` Subcommand

### Requirement: Command Signature

`fab send-keys` SHALL accept two positional arguments: `<change>` and `<text>`. It SHALL resolve `<change>` using standard change resolution (4-char ID, substring, full folder name). It SHALL send the text to the target agent's tmux pane via `tmux send-keys -t <pane> "<text>" Enter`.

#### Scenario: Send text to a known change's pane
- **GIVEN** change r3m7 is active in pane %3
- **WHEN** `fab send-keys r3m7 "/fab-continue"` is executed
- **THEN** the command resolves r3m7 to its worktree pane
- **AND** runs `tmux send-keys -t %3 "/fab-continue" Enter`

#### Scenario: Change not found
- **GIVEN** no change matches "xyz"
- **WHEN** `fab send-keys xyz "hello"` is executed
- **THEN** the command exits non-zero with: `No change matches "xyz".`

### Requirement: Pane Resolution

`fab send-keys` SHALL resolve the target pane by:
1. Resolving the change argument to a folder name via standard change resolution
2. Running `tmux list-panes -a -F "#{pane_id} #{pane_current_path}"` to get all panes
3. For each pane, computing the git worktree root and reading `fab/current` to find the change folder
4. Matching the resolved change folder against pane entries

#### Scenario: Change exists but no tmux pane found
- **GIVEN** change r3m7 exists in `fab/changes/` but no tmux pane has that worktree
- **WHEN** `fab send-keys r3m7 "/fab-continue"` is executed
- **THEN** the command exits non-zero with: `No tmux pane found for change "r3m7".`

#### Scenario: Multiple panes for same change
- **GIVEN** two panes (%3 and %7) both have r3m7 as active change
- **WHEN** `fab send-keys r3m7 "/fab-continue"` is executed
- **THEN** the command sends to the first matching pane
- **AND** prints a warning to stderr: `Warning: multiple panes found for r3m7, using %3`

### Requirement: Tmux Session Guard

`fab send-keys` SHALL check that `$TMUX` is set. If not, it SHALL exit non-zero with: `Error: not inside a tmux session`.

#### Scenario: No tmux session
- **GIVEN** `$TMUX` is unset
- **WHEN** `fab send-keys r3m7 "/fab-continue"` is executed
- **THEN** the command exits non-zero with `Error: not inside a tmux session`

### Requirement: Pane Existence Validation

Before sending, `fab send-keys` SHALL validate the target pane still exists. If the pane is gone, it SHALL exit non-zero with an error.

#### Scenario: Pane disappeared after resolution
- **GIVEN** pane %3 existed during discovery but was closed before send
- **WHEN** `tmux send-keys -t %3` fails
- **THEN** the command exits non-zero with: `Error: pane %3 no longer exists.`

### Requirement: No Idle Check in CLI

`fab send-keys` SHALL NOT enforce idle checks. Policy (check idle before sending) lives in the skill; mechanism (send text to pane) lives in the CLI. This is separation of concerns.

#### Scenario: Send to active agent succeeds at CLI level
- **GIVEN** agent in pane %3 is active (not idle)
- **WHEN** `fab send-keys r3m7 "/fab-continue"` is executed
- **THEN** the command sends the text successfully (no idle guard)
- **AND** returns exit code 0

## Design Decisions

1. **Skill, not package**: The operator requires AI agent interpretation of natural language. It cannot be a shell script or Go binary. The `fab send-keys` CLI primitive is the only binary addition — the coordination logic lives in the skill file.
   - *Why*: Shell scripts can't interpret "rebase k8ds when r3m7 finishes" — they lack the reasoning layer.
   - *Rejected*: Full CLI implementation — coordination requires contextual reasoning.

2. **Pane resolution in send-keys reuses pane-map logic**: Rather than inventing a new resolution path, `send-keys` discovers panes the same way `pane-map` does (tmux list-panes → git root → fab/current). This keeps the resolution consistent.
   - *Why*: Single resolution path reduces bugs and maintenance.
   - *Rejected*: Storing pane IDs in `.status.yaml` — stale pane IDs are dangerous, tmux panes are ephemeral.

3. **No idle check in CLI, policy in skill**: The `send-keys` subcommand is a pure mechanism (send text to pane). The idle check is policy enforced by the operator skill. This matches the separation of concerns pattern in the codebase (e.g., change resolution is in the CLI, usage policy is in skills).
   - *Why*: The CLI shouldn't make policy decisions about when sending is safe.
   - *Rejected*: `--force` flag to bypass idle check — adds complexity for a concern that belongs in the skill.

4. **User-driven, not event-driven**: The operator waits for user input. It does not poll in the background. Notification instructions ("tell me when r3m7 finishes") are held in conversation context and checked on subsequent user interactions.
   - *Why*: A polling daemon would require background process management, which violates the "not a daemon" constraint.
   - *Rejected*: File-watching with inotify — adds external dependencies and daemon-like behavior.
